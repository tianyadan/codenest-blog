# SQL 优化常用技巧

## 1. 避免使用 Select*

- 很多时候我们喜欢使用 select * 来一次性查询表中所有列的数据，但是在实际业务场景中，可能我们真正用到的只有其中一两列，查了很多数据但是不用，**白白浪费数据库资源**，比如:内存或者 cpu。 并且 select * 不会走**覆盖索引**，会出现**大量回表**操作，从而导致查询 sql 的性能极低。

- 改进方法：**只查询具体想要的字段**。`select name,age from users where id =1 ;`

## 2. 用union all代替union

- sql 语句使用 union 关键字后，可以获取重排后的数据，而如果使用 union all 关键字，可以获取所有数据，包含重复的数据。但是排重的过程需要遍历、排序和比较，
它更耗时，更消耗 cpu 资源。所以说能用 union all 就尽量不用union 。

##  3. 小表驱动大表
- 小表驱动大表是指：在做关联查询（JOIN）时，如果有一张表很小，一张表很大，优先从小表开始查，再去大表查，这样可以减少扫描大表的数据量，提高查询效率。
- 举个例子：假设有两张表：用户表 user，只有 **1000** 条数据,订单表 order，有 **100w** 条数据
- 错误示例：大表驱动 ,直接在大表 orders 上查询：
``` 
SELECT o.*
FROM orders o
JOIN user u ON o.user_id = u.id
WHERE u.id IN (1, 2, 3);
```
- 问题：MySQL / PostgreSQL 可能先扫描 orders（100w 条），再做 JOIN。如果 orders 没有针对 user_id 的索引，性能很差。

- **优化示例：小表驱动大表**
- 步骤 1：先查出小表需要的用户 id,步骤 2：再去大表查询,写成子查询:
``` 
SELECT *
FROM orders
WHERE user_id IN (
    SELECT id FROM user WHERE id IN (1, 2, 3)
);
```
- **优点：小表只有 1000 条数据，查询非常快。大表只根据小表的结果去扫描（利用索引 user_id），扫描的数据量小得多。**
- 原则：关联查询中，尽量让查询先走小表，减少大表扫描量。
- **补充 in 适用于左边大表，右边小表。exists 适用于左边小表，右边大表**

## 4. 批量操作
- 不要在业务中频繁用 for 循环遍历，然后单条插入数据库，要用数据库的批量插入功能，能减少性能开销！！

## 5. 多用 limit
- 有时我们需要查询某些数据的第一条数据，举例：查询某个用户下的第一个订单，先看看他第一次的首单时间。如果用：
```
select id,create_date from order where user_id = 123 order by create_date asc;
```
得到一个集合，然后再代码中获取第一个元素的数据，即首单的时间。 **这样做效率非常不高，需要先查询所有数据，浪费资源。**

**正确做法是：**

```
select id,create_date from order where user_id = 123 order by create_date asc limit 1;
```
使用 `limit1` ,只返回该用户下单时间最小的那一条数据即可。

- 另一个例子：**SQL 查找是否存在，别再使用 count 了！**
 
如果想查询有没有该数据，都只是存不存在，不关心多少条，只需要用 limit 限制，查询 1 条即可，这样数据库查到一条记录就返回，说明该数据存在。大大节省资源。

## 13. 优化索引
阿里巴巴开发手册规定：单表的索引数量尽量控制在`5`个以内，并且单个索引中字段数不宜超过`5`个。mysql 会使用 B+树来保存索引，在 insert、update、delete 操作时，需要更新B+树索引，如果索引过多会带来额外的性能。
