# 微服务架构下的安全实战：一次SQL注入攻击的复盘与防御

# 事件起因： 

**2026-04-21 早上测试功能发现报错，于是想去查看异常日志**

作为一个微服务架构的维护者，我习惯于定期检查服务的运行状态。然而，在查看 exam-service 的日志时，一连串密集的异常堆栈引起了我的注意。

不同于往常的业务报错，这些日志呈现出一种“机械化”的规律：在极短的时间内（毫秒级），同一个接口被高频访问，且伴随着大量的 `MethodArgumentTypeMismatchException`（方法参数类型不匹配异常）。

直觉告诉我，这不是正常的业务流量，而是一次针对系统的自动化攻击。

## 分析过程：抽丝剥茧，还原真相

为了搞清楚状况，我登录到服务器，通过 `docker exec` 进入了容器内部，直接查看了 `/app/logs` 下的原始日志文件。

1 ）**攻击特征识别**：

通过分析日志中的请求参数，我发现攻击者向 id、year 等本应为数字类型的参数中，注入了大量包含 SQL 关键字的字符串，例如：

- `' union select 1--` 
- `2026" union select md5(3141592657)--`  # 测试数据库是否执行函数
- `4 union select 1,2--` # 测试字段数量（列数）

这些是典型的 SQL注入（SQL Injection） 攻击载荷。特别是 `md5(3141592657)` 这种特殊的探测语句，带有明显的自动化扫描工具（如 SQLMap）的指纹特征。

2 ）**攻击原理分析**
攻击者的意图非常明显：他们试图利用 SQL 的 UNION 操作符，将恶意查询拼接到原本的数据库查询中，从而窃取数据库结构或敏感数据。
他们利用的是“报错注入”的原理：通过输入单引号 `'` 或双引号 `"` 来破坏原有的 SQL 语句结构，观察系统的报错信息来判断是否存在漏洞。

3 ） **为什么攻击没有成功？**

这次攻击之所以没有造成数据泄露，主要归功于两层防御：

- 框架层的类型安全：我的后端接口定义参数为 Integer 或 Long 类型。当 Spring Boot 接收到 `' union select...` 这种字符串时，在尝试将其转换为数字时直接抛出了 NumberFormatException。攻击载荷在接触到数据库层之前，就已经被应用层的类型检查拦截了。

- 网关层的流量清洗：虽然日志看起来很多，但实际上大部分高频流量已经被 Gateway 层的 Sentinel 限流规则拦截。我看到的日志，仅仅是“漏网之鱼”。

## 处理结果：有惊无险

经过排查，确认系统未出现数据泄露，数据库连接正常，业务功能未受影响。

- 攻击源：全网扫描的自动化僵尸网络（Botnet）。

- 受损情况：无。仅产生了少量的垃圾日志。

- 当前状态：系统持续稳定运行，攻击行为仍在继续，但被有效防御。

这次事件虽然未造成损失，但也暴露了我们在安全监控和防御粒度上的不足。

# 后续优化：构建纵深防御体系

为了将“被动防御”转变为“主动免疫”，我制定了以下三个维度的优化方案：

1 ） **感知层：建立实时告警机制**
痛点：目前只能通过翻看日志发现攻击，滞后性强。
改进：升级 GlobalExceptionHandler。当捕获到参数类型异常且内容包含 SQL 关键字（如 union, select, drop）时，判定为恶意攻击。
动作：触发异步告警，通过钉钉/企业微信机器人推送攻击者的 IP、URL 和攻击载荷，实现秒级感知。

2 ） **防御层：精细化限流**
痛点：现有的限流是基于整体 QPS 的，不够精准。
改进：利用 Sentinel 的热点参数限流功能。
策略：针对 id、userId 等关键参数进行监控。如果某个参数的值频繁触发异常，或者包含特殊字符，直接在网关层进行拦截并封禁该 IP，不再让其进入业务服务

3 ） **代码层：彻底杜绝隐患**
痛点：虽然框架挡住了攻击，但不能依赖“运气”。
改进：
全面审查：全局搜索代码中的 SQL 拼接（如 MyBatis 的 ${}），全部替换为预编译语句（#{}）。
参数校验：在 Controller 层引入 @Valid 和正则表达式校验，对于 ID 类参数，严格限制只能输入数字，从源头拒绝非法字符。

# 总结

<span class="md-inline-color md-inline-color--e74c3c">网络安全是一场没有终点的博弈。这次事件提醒我，“默认安全”的设计原则至关重要。即使没有专门配置复杂的 WAF，良好的编码习惯（如强类型约束）和架构设计（如网关限流）也能在关键时刻成为保护系统的坚实盾牌。
</span>
