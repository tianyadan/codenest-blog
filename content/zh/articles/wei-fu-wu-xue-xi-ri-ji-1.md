# 微服务学习日记 1

## 一、微服务理念

- 是一种软件架构风格，每个微服务都围绕特定的业务构建，通过明确定义的 API 进行通信。
- 微服务不是银弹，不是什么时候都都适合。

#### 微服务作用：
- 独立部署与运行
- 按业务功能划分
- 技术选型灵活
- 弹性扩展
- 故障隔离

#### 微服务弊端
- 分布式复杂性
- 运维复杂度提升
- 服务治理挑战
- 数据一致性难题

## 二、微服务架构设计方案

改造微服务之前先考虑的问题。

1. 设计上：怎么划分服务？ 按照什么原则将单体应用拆分为多个微服务？
2. 实现上：用什么技术？ 选择呢什么技术栈来支撑微服务架构
3. 治理上：如何管理服务？如何实现服务注册发现、配置管理、监控等。

### （一）服务划分：
**服务划分是整个改造过程中最关键的一步，拆分得好不好直接决定了后续开发和维护的复杂度。** 

**拆分原则**：

1. 业务领域驱动：按照业务功能进行拆分，确保每个服务都有明确的业务边界。比如用户管理、应用管理，每个领域业务相对独立。

2. 数据一致性：相关性强的数据保持在同一服务中，减少跨服务数据依赖，比如用户信息和用户权限放在同一个服务中，防止跨服务查询。

3. 性能考虑：将重新模块独立出来，便于单独优化和扩展。

4. 扩展性需求：根据不同模块进行拆分，访问频率高的模块可以独立扩展，不影响其他服务。

**特别说明：不能拆的不要硬拆。**

**通用模块：**

common 模块：主要包含所有服务公用的代码，如异常处理、工具类、常量定义等，为其他服务提供基础设施支持。

model: 统一的数据模型定义，包含实体类，DTO、VO、枚举类等，确保各服务间数据格式的一致性。

client: 定义需要内部调用的服务接口，作为服务间通信的契合，实现服务间松耦合。

**业务服务：**

- 用户服务：作为整个系统的基础，同意管理用户状态和权限。负责用户注册、登录、注销、权限校验等核心功能。所有业务都需要用户信息，所以用户服务是其他服务的基础依赖。

- 应用服务：业务核心服务。 负责应用的完整生命周期管理。

- 其他服务等等...

**最后，需要整理一个微服务划分表 ，包括各个服务的职责，端口分配，路由规划和依赖关系，便于后续的开发实现。**

示例：

| 服务名称 | 端口和路由前缀 |主要功能 | 依赖服务 |
| --- | --- |--- |--- |
| 通用模块 |   ||  | 
| common |  - |注解、异常处理、工具类、常量、公共响应类| - | 
| model |  - |实体类、DTO、VO、枚举类、AI 模型类| common |
| client |  - |服务接口定义、内部调用契约| common\model |
| 业务服务 |   ||  | 
| user |  8124 端口，/api/user/** |用户管理、权限认证、用户信息维护| Redis\Mysql | 
| app |  8125 端口，/api/app/** |应用管理、相关业务流程| Redis\Mysql\用户服务等 | 

### （二）微服务技术选型：

微服务需要解决服务拆分、服务间通信、服务发现、统一网关等核心问题。对应这些需求，我们需要选择合适的技术栈，包括开发框架、注册中心、服务调用、微服务网关。 推荐使用 SpringCloud Alibaba ， Nacos 、Dubbo 、Higress 网关这套完整的解决方案。

**注册中心：**

注册中心是微服务架构的核心组件，它解决了服务发现问题。类似于一个电话簿，服务启动时向注册中心报告自己的地址信息，服务调用时从注册中心查询目标服务地址。

**微服务调用**

- HTTP/REST ： 基于 HTTP 协议的 RESTful API 调用 ，简单易用，跨语言支持好。
- RPC：远程过程调用，如 Dubbo，gRPC，性能更高，但通常需要相同的技术栈。
- 消息队列：如 RocketMQ ，Kafka ，支持异步调用，适合解耦场景。

**微服务网关**

- SpringCloud Gateway ：Spring 官方出品 ，和 Spring Cloud 生态集成度高。
- Higress：阿里巴巴开源的云原生 API 网关，基于 Enovy ，性能优异。
- kong：基于 Nginx 的开源网关，插件丰富
-  Zuul：Netflix 开源，性能一般
