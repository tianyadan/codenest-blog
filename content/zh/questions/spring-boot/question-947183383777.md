## 答案

## 核心要点：

**基于 Jakarta EE**：
- SpringBoot 3.x 迁移到了 jakarta EE ，这意味着一些核心包名从`javax.*` 变更成 `jakarta.*` 。 也是从 Java EE 迁移到 jakarta EE 后的重大变化，开发时需要注意兼容性问题。
- SpringBoot 2.x 仍然基于 java EE （即 `javax.*` 包）

**JDK 17 支持：**

 - SpringBoot 3.x 要求使用 jdk17 为最低版本，它充分利用了 jdk17 的新特性，包括性能优化和语言增强。
 - SpringBoot 2.x 支持 jdk 8、11、17 兼容范围更广。

**原生编译支持(GraalVM Native Image)：**

- SpringBoot 3.x 提供了对 GraalVM Native Image 的开箱即用支持，可以将 Spring 应用编译成本地可执行文件，显著减少启动时间和内存专用，特别适合微服务和云原生应用。**毫秒级别启动**，**更低的内存占用**
- SpringBoot 2.x 没有原生编译的内置支持，虽然可以通过一些额外配置使用 GraalVM ，但支持有限。

**改进的观察性(Observability):**

- SpringBoot 3.x 将 Micrometer 和 Micrometer Tracing 深度集成，引入了更完善的分布式追踪、日志关联和性能指标收集，支持 OpenTelemetry 标准。Observability 使开发者能够更好地监听和分析应用的运行状况。

- SpringBoot 2.x 支持基本的监控和追踪，但没有统一的 Observability 框架。

**安全性增强：**

- SpringBoot 3.x 增强了对 Spring Security 6 的支持，进一步强化了身份认证、授权和安全配置的能力，特别是对于 OAuth 2.1 、 openID Connect 等标准的支持更加完善。
- Spring Boot 2.x 支持 Spring Security 5，在安全性上略有局限性，特别是在 OAuth 和 JWT 的使用场景下。

**模块化和简化的依赖管理：**
- SpringBoot 3.x  对内部依赖进行了一些模块化调整，清理了一些不再使用或过时的库，优化了依赖管理，使得应用更加轻量和便于扩展
- SpringBoot 2.x 的依赖库和自动配置虽然强大，但是有时显得过于庞大，存在一些历史遗留的依赖和配置
