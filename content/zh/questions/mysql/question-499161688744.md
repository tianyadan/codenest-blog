## 答案

索引下推是 **MySQL 5.6** 引入的优化技术，能减少回表次数，提升查询效率。

核心思路是把查询条件<span style="color:#e67e22">从 Server 层</span>下推到<span style="color:#f39c12">存储引擎层</span>，在引擎层就把不符合条件的数据过滤掉，<span style="color:#e74c3c">不再回表</span>查询完整数据。

以前的流程：引擎层用索引定位完数据，返回主键给 Server 层，Server 层回表拿到完整数据，再用剩余条件过滤，**有了索引下推，引擎层可以直接用索引里的列过滤，不符合条件的压根不回表，大大减少 I/O** 。

索引下推主要用在联合索引上。

**举个例子：**

假设有一个用户表 **users**，包含以下列
- id (主键)
- name
- age
- city

并且我们创建了如下联合索引：


```sql
CREATE INDEX idx_name_age_city ON users(name, age, city);
```

现在，如果你执行以下查询：


```sql
SELECT id, name, age, city
FROM users
WHERE name = 'Alice' AND age > 30 AND city = 'Shanghai';
```

**没有索引下推的情况**：

1. 存储引擎首先通过联合索引 idx_name_age_city 定位到符合 name = 'Alice' 和 age > 30 的记录。
2. 存储引擎返回这些记录的主键（id）。
3. 然后，Server 层使用这些主键回表，拿到完整的记录数据。
4. 最后，Server 层再根据条件 city = 'Shanghai' 对记录进行过滤。


**有了索引下推的优化：**

1. 存储引擎在定位数据时，不仅根据 name = 'Alice' 和 age > 30 查找符合条件的记录，还会将 city = 'Shanghai' 这个条件也下推到存储引擎层。
2. 存储引擎直接返回符合 name = 'Alice'、age > 30 和 city = 'Shanghai' 的记录，不需要再回表。
3. 因此，数据只会在存储引擎层进行筛选，不再有额外的回表操作，减少了 I/O 开销。
        
<span style="color:#e74c3c">通过这个优化，MySQL能够在不返回不必要数据的情况下，直接在存储引擎层完成更多的过滤工作，从而提升查询效率，特别是在大量数据和复杂查询的情况下。</span>
