## 答案

MySQL事务隔离级别共有**4**种，从低到高依次为：**读未提交**、**读已提交**、**可重复读**、**串行化**。MySQL InnoDB默认使用可重复读级别，而互联网公司普遍采用读已提交级别以提高性能。

**事务隔离级别详解**

1.读未提交 (READ UNCOMMITTED)

- 事务可以看到其他事务未提交的数据，隔离性最低。
- 存在**脏读**问题：读取的数据**可能被回滚**，导致数据不一致。
- 实际生产环境极少使用，仅可能用于对数据一致性要求不高的离线统计任务。

2.读已提交 (READ COMMITTED == RC)


- 事务只能看到其他事务已提交的数据，**解决了脏读问题**。
- 存在**不可重复读问题**：**同一事务内两次查询同一数据可能结果不同**。
Oracle、PostgreSQL、SQL Server默认使用此级别。
互联网公司普遍采用此级别，原因包括：减少undo log版本链内存消耗、缩小锁范围避免死锁、大多数业务场景不需要"同一事务内数据不变"的保证。

3.可重复读 (REPEATABLE READ == RR)

- MySQL InnoDB默认级别，通过MVCC实现。
- 事务开始后看到的数据是开始时的快照，**解决了不可重复读问题**。
- 标准SQL定义下仍**存在幻读问题**（**范围查询返回行数不一致**）。
- InnoDB通过快照读（MVCC）和当前读（间隙锁）缓解幻读问题，但未完全消除。

4.串行化 (SERIALIZABLE)

最严格隔离级别，相当于事务串行执行，**彻底解决所有并发问题**。
InnoDB会将普通SELECT自动转为SELECT ... FOR SHARE（共享锁），导致读写排队。
**性能影响严重，实际项目很少使用**。

**MySQL默认使用可重复读的原因**

历史原因：早期MySQL主从复制使用Statement格式binlog，可重复读保证事务内数据一致，避免主从不一致。
虽然现在有Row格式binlog支持读已提交级别，但MySQL未更改默认值。
