## 答案

**优点：**

1 ) **减少 I/O 操作**。查询直接从索引中拿数据。避免了访问主键索引的数据页。如果结果集大，省掉的回表次数可能是几万几十万次。

**例子：**

 表结构：
```sql
CREATE TABLE user (
  id INT PRIMARY KEY,
  name VARCHAR(50),
  age INT,
  city VARCHAR(50),
  INDEX idx_age_name(age, name)
);
```

查询：


```sql
SELECT name FROM user WHERE age = 20;
```

用到了**联合索引 (age, name)**，不需要回表查主键索引。如果有 10 万条数据 ，回表 = 10 万次 I/O ，覆盖索引 = 0 次回表 。

2 ）**查询更快。** 索引比表数据更紧凑，存储信息少，读取速度更快。而且索引数据在内存里的命中率通常比表数据高。

**例子：**

表：


```sql
CREATE TABLE order_info (
  id INT PRIMARY KEY,
  user_id INT,
  amount DECIMAL,
  create_time DATETIME,
  remark TEXT,
  INDEX idx_user_id_amount(user_id, amount)
);
```

查询：


```sql
SELECT user_id, amount FROM order_info WHERE user_id = 100;
```

索引里只存：user_id + amount + 主键id ，表数据里还有：remark（大字段） + 其他字段。

索引页更小、更紧凑，一页（16KB）：表数据可能只能放 10 条，索引可能能放 100 条。读取同样 1000 条数据： 表：需要 100 页 ，索引：只需要 10 页 。

**索引数据更紧凑，同样的 I/O 可以读取更多记录，从而查询更快。**


3 ）**减少内存占用。** 只需要把索引页加载到 Buffer Pool，不用加载完整的数据页，内存利用率更高。

**例子：**

还是上面 order_info 表：假设
- 表数据：1GB
- 索引：100MB
- Buffer Pool：200MB

查表情况：


```sql
SELECT * FROM order_info WHERE user_id = 100;
```

需要加载：索引页 + 数据页 ，数据页很大 → 很快被淘汰 。

覆盖索引：


```sql
SELECT user_id, amount FROM order_info WHERE user_id = 100;
```
只需要加载： 索引页（100MB 能装下） 。
效果：索引几乎可以常驻内存 ， 命中率极高 。
