## 答案

二级索引本身**没有 MVCC 快照**。

原因是二级索引条目只存了索引列值和主键值，不包含 InnoDB 的隐藏列 trx_id 和 roll_ptr 。 没有这两个字段，就没办法判断这条记录属于哪个事务版本，在 Undo Log 里往哪找历史版本 。

所以当二级索引项被修改或者**打了删除标记时**，InnoDB 必须**回表到聚簇索引**，通过聚簇索引上的trx_id 和 undo log 版本链来判断当前事务能看到哪个版本。

一个容易踩的坑是就是查询走了覆盖索引，**只要涉及到版本判断，覆盖索引也会失效**，还得回表。

不过，也不是每次都会需要回表，二级索引每个页上记录了最后修改这个索引页的最大事务 ID，**当前事务启动时的最小活跃事务 ID 大于最大事务 ID** 则不用回表。

**扩展知识：**

**InnoDB 对覆盖索引的优化：**

虽然二级索引条目没有 trx_id ，但 InnoDB 并不是每次傻傻地回表。它在二级索引的每个页记录了一个page_max_trx_id ，表示最后修改这个索引页的最大事务 ID。查询的时候会先看这个 page_max_trx_id ：如果它比当前事务启动时的最小活跃事务 ID 还小，说明这个页上所有记录对当前事务都是可见的，直接用二级索引的的数据就行，不用回表。

只有当 page_max_trx_id 太新，或者记录被打了删除标记，才需要回表到聚簇索引确认可见性。这个优化在高并发读多写少的场景下效果明显，能省掉大量不必要的回表操作。

**实际场景：**

假设有这么张表和索引：


```sql
CREATE TABLE t (
    id INT PRIMARY KEY,
    name VARCHAR(20),
    age INT,
    INDEX idx_name_age(name, age)
);
```

事务 A 执行更新但是没提交：


```sql
-- 事务 A
UPDATE t SET age = 20 WHERE name = 'Tom';
```

事务 B 来查询：


```sql
-- 事务 B
SELECT age FROM t WHERE name = 'Tom';
```

虽然 idx_name_age  是覆盖索引，理论上不用回表，但是因为事务 A 改了 age 还没提交，二级索引页的 page_max_trx_id 变新了，InnoDB 没法直接判断 age = 20 对事务 B 是否可见，只能回表到聚簇索引，通过 trx_id 和 undo log 找到事务 B 能看的那个版本。


**二级索引更新与聚簇索引更新的区别：**

聚簇索引记录是原地更新的，旧值通过 roll_ptr 串到 undo log 里形成版本链。
二级索引不一样，它不会原地更新。当二级索引列被改了，老记录会被打上删除标记，同时插入一条新纪录。被标记删除的记录等着 purge 线程来清理。

这么设计是因为二级索引没有 roll_ptr ，没法像聚簇索引那样串版本链。只能用标记删除+新增的方式来保留历史版本，让别的事务还能看到老数据。
