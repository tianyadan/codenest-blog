## 答案

**HashMap 是一个基于“数组 + 链表 + 红黑树”的 key-value 存储结构，通过 hash 算法快速定位数据位置。**

你可以把它理解成： 数组负责“快速定位桶”，链表/红黑树负责“解决 hash 冲突”。

**1 ） HashMap 底层结构**

JDK 8 之后，HashMap 底层是：

```java
Node<K,V>[] table;
```

每个数组位置叫一个 桶 bucket,每个桶里面可能是：

```java
数组

 ↓

[0] null

[1] Node -> Node -> Node

[2] null

[3] TreeNode 红黑树

[4] Node
```

也就是：<span class="md-inline-color md-inline-color--e74c3c">HashMap = 数组 + 链表 + 红黑树</span>

**2 ）put 的流程**

比如：`map.put("name", "tian");`

大体流程是：

第一步：计算 key 的 hash 值 ：`hash = hash(key)`，HashMap 不会直接使用 key.hashCode()，而是会做一次扰动计算： `(h = key.hashCode()) ^ (h >>> 16)` ，目的是让高 16 位也参与运算，减少 hash 冲突。

第二步：根据 hash 定位数组下标 ：HashMap 数组长度默认是 16，并且始终是 2 的幂，下标计算：`index = (n - 1) & hash;` 不是用 % 取模，而是用位运算 & ，因为当数组长度是 2 的幂时：`hash % n` 等价于 `hash & (n - 1)` ，位运算更快。

**3 ） hash 冲突是什么？**

不同的 key 可能算出来同一个数组下标 ，比如：`key1 -> hash -> index = 3` ，`key2 -> hash -> index = 3` ， `key3 -> hash -> index = 3` ，**这就叫 hash 冲突**。

HashMap 解决冲突的方式：<span class="md-inline-color md-inline-color--e67e22">数组某个位置上挂链表</span>，JDK 8 之后，如果链表太长，会转换成红黑树。

**4 ） 链表什么时候变红黑树？**

满足两个条件才会树化：

- 链表长度 >= 8
- 数组长度 >= 64

如果链表长度到了 8，但是数组长度还小于 64，HashMap 会优先扩容，而不是树化。原因很现实：**冲突多可能不是 key 太多，而是数组太小，先扩容更划算。**

**5 ） 红黑树什么时候退化成链表？**

当红黑树节点数量减少到 6 左右时，会退化回链表，为什么不是 8？
- 因为如果树化阈值是 8，退化阈值也是 8，就会频繁在链表和树之间切换。

所以：树化：8 ， 退化：6 中间留了缓冲区。

**6 ） get 的流程**

比如：`map.get("name");`

流程是：

```text
1. 计算 key 的 hash
2. 根据 hash 算数组下标
3. 找到桶位置
4. 如果第一个节点就是目标，直接返回
5. 如果是链表，遍历链表
6. 如果是红黑树，按树查找
```

查找时会比较：hash 是否相等 、key 是否 ==  、key.equals() 是否相等

**所以 HashMap 判断 key 是否一样，不只是看 hash，还会用 equals**。

**7 ） 为什么重写 equals 必须重写 hashCode？**

因为 HashMap 先用 hashCode 定位桶，再用 equals 判断 key 是否相等，如果你只重写 equals，不重写 hashCode，可能出现：`a.equals(b) == true` 但是 `a.hashCode() != b.hashCode()` ，结果就是：两个逻辑上相等的对象，被放到了不同桶里 ，HashMap 就找不到正确数据了。

**8 ） 默认容量和负载因子**

HashMap 默认容量：<span class="md-inline-color md-inline-color--e74c3c">16</span> ， 默认负载因子：<span class="md-inline-color md-inline-color--e74c3c">0.75</span> ， 扩容阈值：`threshold = capacity * loadFactor` ， 默认情况下：16 * 0.75 = 12 。也就是说，当元素数量超过 12 时，HashMap 会扩容。

**9 ） 为什么负载因子是 0.75？**

这是一个折中值。如果负载因子太小，比如 0.5：冲突少，查询快，但浪费内存 ；如果负载因子太大，比如 1.0：节省内存，但冲突变多，查询变慢 。所以 0.75 是时间和空间的平衡。

**10 ） HashMap 怎么扩容？**
 
HashMap 扩容时，数组容量变成原来的 2 倍：`16 -> 32 -> 64 -> 128` ，JDK 8 做了优化 ，扩容后，原来桶里的节点只会有两个去向：

- 原位置
- 原位置 + oldCap

比如原来容量是 16，某个节点在下标 5。

扩容到 32 后，它要么还在：5 ，要么去：5 + 16 = 21 。这是因为数组长度是 2 的幂，扩容只多看 hash 的一位。

**11 ） 为什么 HashMap 容量必须是 2 的幂？**

主要原因有两个：

- 第一，方便快速取下标 ： `index = (n - 1) & hash;` 比 % 取模更快。
- 第二，扩容迁移更高效 ：扩容后节点位置只需要判断 `(hash & oldCap) == 0` ，如果等于 0，位置不变 ，否则移动到：原下标 + oldCap ，这就是 JDK 8 HashMap 扩容很巧妙的地方。

**12 ） HashMap 线程安全吗？**

**不安全** ，多线程同时 put 时可能出现 ：

- 数据覆盖
- 数据丢失
- 扩容异常
- 读取到脏数据

JDK 7 里并发扩容甚至可能出现链表成环问题，JDK 8 虽然修复了链表头插法导致的成环风险，但 HashMap 依然不是线程安全的。多线程环境应该使用：`ConcurrentHashMap`

**13 ） JDK 7 和 JDK 8 HashMap 的区别**

重点区别：JDK 7：数组 + 链表 ； JDK 8：数组 + 链表 + 红黑树
插入方式：JDK 7：头插法；JDK 8：尾插法
扩容问题：JDK 7：并发扩容可能导致链表成环；JDK 8：优化了扩容逻辑，但仍然线程不安全
性能：JDK 8 在 hash 冲突严重时，可以从链表 O(n) 优化到红黑树 O(log n)
