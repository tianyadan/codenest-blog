## 答案

## 简短答案
RAG（Retrieval-Augmented Generation，检索增强生成）即检索增强生成，是一种将知识检索与大语言模型结合的技术方案。
其核心思想是：
> 先从知识库中检索与问题相关的内容，再将检索结果作为上下文提供给大模型，由大模型生成最终答案。
RAG 能够解决大模型知识过时、幻觉严重、无法使用企业私有知识等问题。
---
# 什么是 RAG？
## 概念解释
传统大模型的知识来源于训练阶段的数据。
例如：
- GPT
- DeepSeek
- Qwen
- Llama
模型训练完成后，其参数基本固定。
如果用户提问：
> 公司内部报销流程是什么？
或者：
> 某产品最新版本的功能有哪些？
大模型通常无法准确回答。
因为这些内容并未参与模型训练。
此时就需要 RAG。
RAG 的工作方式：
```text
用户问题
    ↓
知识库检索
    ↓
找到相关文档
    ↓
构建 Prompt
    ↓
大模型生成答案
```
本质上：
```text
LLM 负责理解和生成
RAG 负责查找资料
```
可以理解为：
```text
大模型 = 聪明的大脑
RAG = 外部知识库
```
传统大模型：
```text
靠记忆回答问题
```
RAG：
```text
先查资料再回答问题
```
---

# 为什么需要 RAG？
传统大模型存在以下问题：
## 1、知识存在时效性
例如：
```text
今天发生的新闻
公司昨天发布的新制度
最新产品说明书
```
模型训练时并不知道这些内容。

---
## 2、容易产生幻觉
模型会根据概率预测下一个词。
因此可能出现：
```text
一本不存在的书
一个不存在的接口
一个不存在的 API
```
即所谓的幻觉（Hallucination）。

---
## 3、无法访问企业私有数据
例如：
```text
企业内部文档
运维手册
工单知识库
产品说明书
数据库设计文档
```
这些内容不会出现在公开训练数据中。

RAG 可以让模型在回答前实时检索这些资料。

# RAG 的主要流程
面试重点记忆：
```text
数据准备
↓
向量化
↓
存入向量数据库
↓
用户提问
↓
问题向量化
↓
相似度检索
↓
构建 Prompt
↓
LLM 生成答案
```
---
## 第一步：知识库数据准备
收集企业内部数据：
```text
Word
PDF
Excel
Markdown
数据库记录
网页内容
FAQ
```
例如：
```text
员工手册.pdf
产品文档.docx
运维手册.md
```
---
## 第二步：文档切片（Chunk）
大模型无法一次读取超长文档。
因此需要切片。
例如：
原文：
```text
SpringCloud 全部教程
```
切分为：
```text
Chunk1：Nacos介绍
Chunk2：服务注册流程
Chunk3：配置中心原理
Chunk4：负载均衡实现
```
常见切片大小：
```text
200~1000 Token
```
---
## 第三步：Embedding 向量化
将文本转换为向量。
例如：
```text
"Nacos是什么"
```
转换为：
```text
[0.12, 0.55, 0.33, ...]
```
向量能够表示文本语义。
常见 Embedding 模型：
- BGE
- BGE-M3
- text-embedding-3-small
- text-embedding-3-large
- Qwen Embedding

---
## 第四步：存入向量数据库
将向量保存到向量数据库。
常见向量库：
- Milvus
- Elasticsearch Vector
- Weaviate
- Chroma
- PGVector
- FAISS
存储结构：
```text
文本内容
+
向量
+
元数据
```
例如：
```json
{
  "content":"Nacos是阿里开源的注册中心",
  "vector":[0.13,0.22,0.45],
  "source":"nacos.pdf"
}
```

---
## 第五步：用户提问
用户输入：
```text
Nacos有什么作用？
```
---
## 第六步：问题向量化
同样调用 Embedding 模型：
```text
Nacos有什么作用？
```
变成：
```text
[0.16,0.42,0.67...]
```
---
## 第七步：向量检索
利用相似度算法查找最相关内容。
常见算法：
- Cosine Similarity（余弦相似度）
- 欧氏距离
- 点积
例如：
```text
Top1：
Nacos是注册中心
Top2：
Nacos支持配置中心
Top3：
Nacos支持服务发现
```
返回最相关的几个 Chunk。
---
## 第八步：构建 Prompt
将用户问题与检索结果拼接。
例如：
```text
你是一名技术专家。
已知资料：
Nacos是阿里开源的注册中心。
Nacos支持配置中心。
Nacos支持服务发现。
用户问题：
Nacos有什么作用？
```
---
## 第九步：LLM 生成答案
最终发送给：
- GPT
- DeepSeek
- Qwen
模型根据检索结果生成答案：
```text
Nacos主要具备服务注册发现和配置管理功能，
能够帮助微服务实现服务治理。
```

# 面试回答
RAG（Retrieval-Augmented Generation）即检索增强生成，是当前企业级 AI 应用最主流的架构之一。
它的核心思想是在大模型生成答案之前，先从外部知识库中检索相关内容，再将检索结果作为上下文提供给大模型，从而提升回答准确率并减少幻觉。
RAG 的典型流程包括：
1. 文档采集
2. 文档切片（Chunk）
3. Embedding 向量化
4. 存入向量数据库
5. 用户提问
6. 问题向量化
7. 向量相似度检索
8. TopK 文档召回
9. Prompt 构建
10. LLM 生成最终答案
目前很多企业知识库问答系统、客服机器人、智能运维、智能工单系统、Dify 工作流等产品底层都采用了 RAG 架构。

# 面试官可能的追问

## 追问1：为什么要进行文档切片？

### 答案
原因有两个：
1. 大模型上下文长度有限
2. 检索粒度更细
如果整本 PDF 直接检索：
```text
100页文档
```
可能会返回大量无关内容。
切片后：
```text
一个 Chunk 约 300~800 Token
```

检索准确率更高。

---

## 追问2：Embedding 是什么？
### 答案
Embedding 是将文本转换成高维向量的过程。
例如：
```text
Redis是什么？
```
转换为：
```text
[0.12,0.35,0.76...]
```
向量能够表示文本语义。
语义越接近：
```text
Redis是什么
Redis有什么作用
```
两个向量距离越近。

因此可以进行相似度检索。

---
## 追问3：RAG 和微调（Fine-Tuning）有什么区别？
### 答案
| 对比项 | RAG | Fine-Tuning |
|-|-|-|
| 更新知识 | 快 | 慢 |
| 训练成本 | 低 | 高 |
| 是否重新训练模型 | 否 | 是 |
| 数据更新 | 实时 | 需重新训练 |
| 企业知识库场景 | 非常适合 | 不适合 |
RAG 本质是：
```text
查资料再回答
```
Fine-Tuning 本质是：
```text
把知识学进模型参数
```
企业知识库场景通常优先选择 RAG，因为更新成本低且实时性更强。
