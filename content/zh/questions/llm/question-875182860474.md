## 答案

## 简短答案
A2A（Agent2Agent）协议是一种用于 **Agent 与 Agent 之间通信协作** 的开放协议。
它的核心工作原理可以概括为：
**Agent 发现 → 能力描述 → 发起任务 → 任务执行 → 状态同步 → 返回结果**
简单来说：
> Agent 不再直接调用另一个 Agent 的代码，而是通过统一协议发现对方能力，然后像调用 API 一样发起任务请求，最终获得执行结果。
---
# 什么是 A2A？
A2A 全称：
```text
Agent2Agent Protocol
```
即：
```text
智能体与智能体通信协议
```
如果说：
```text
HTTP
解决浏览器和服务器通信
MCP
解决Agent与工具通信
A2A
解决Agent与Agent通信
```
那么 A2A 的目标就是：
```text
让不同厂商
不同框架
不同语言
开发出来的Agent
能够相互协作
```
例如：
```text
Google ADK Agent
↓
调用
↓
LangGraph Agent
↓
调用
↓
CrewAI Agent
```

理论上都能互通。

---
# 为什么需要 A2A？
假设你开发了一个 AI ERP 系统。
里面有：
```text
采购Agent
库存Agent
财务Agent
审批Agent
```
用户说：
```text
帮我采购100台服务器
```
这时：
```text
采购Agent
```
无法独立完成。
它需要：
```text
查询库存
↓
申请预算
↓
发起审批
```
于是需要：
```text
库存Agent
财务Agent
审批Agent
```
共同参与。
这就产生了：
```text
Agent ↔ Agent
```

协作需求。

---
# A2A 的完整工作流程
一个标准 A2A 请求通常经过六个阶段：
```text
Agent发现
    ↓
能力获取
    ↓
任务创建
    ↓
任务执行
    ↓
状态同步
    ↓
结果返回
```
---

# 第一阶段：Agent Discovery（Agent发现）
首先需要知道：
```text
网络中有哪些Agent
```
类似于：
```text
服务注册中心
```
概念。
A2A 提供：
```text
Agent Card
```
机制。
每个 Agent 都会公开自己的信息。
例如：
```json
{
  "name":"Finance Agent",
  "description":"负责财务分析",
  "skills":[
    "budget_analysis",
    "cost_report"
  ]
}
```

其它 Agent 可以读取。

---
# 什么是 Agent Card？
Agent Card 可以理解为：
```text
Agent身份证
+
Agent说明书
```
类似于：
```text
Swagger
OpenAPI
服务注册信息
```
作用是：
告诉别人：
```text
我是谁
我能干什么
如何调用我
```
例如：
```json
{
  "name":"HR Agent",
  "description":"招聘专家",
  "skills":[
    "resume_review",
    "interview_design"
  ]
}
```

---
# 第二阶段：Capability Discovery（能力发现）
找到 Agent 后。
需要知道：
```text
它会什么
```
例如：
```text
HR Agent
```
支持：
```text
简历筛选
面试题生成
人才评估
```
而：
```text
Finance Agent
```
支持：
```text
财务分析
预算评估
```

这样调用方才能决定是否合作。

---
# 第三阶段：Task Creation（任务创建）
调用 Agent 发起任务。
例如：
```text
生成2026年招聘预算
```
A2A 请求：
```json
{
  "task":"generate_budget",
  "input":{
      "headcount":50
  }
}
```
发送给：
```text
Finance Agent
```
---

# 第四阶段：Task Execution（任务执行）
被调用 Agent 开始工作。
例如：
```text
读取预算数据
↓
分析成本
↓
计算结果
↓
生成报告
```
任务可能：
```text
1秒完成
1分钟完成
1小时完成
```
---

# 第五阶段：状态同步
这是 A2A 一个非常重要的特点。
很多任务不是立即完成。
例如：
```text
市场分析
代码生成
深度研究
财务审计
```
可能执行很久。
因此 A2A 支持：
```text
Task Status
```
例如：
```json
{
  "status":"running",
  "progress":"50%"
}
```
或者：
```json
{
  "status":"completed"
}
```

调用方能够实时知道执行情况。

---
# 第六阶段：Artifact 返回结果
执行完成后。
返回结果。
A2A 中称为：
```text
Artifact
```
即：
```text
任务产物
```
例如：
```json
{
  "status":"completed",
  "artifact":{
      "report":"预算分析报告..."
  }
}
```

返回给发起 Agent。

# A2A 的核心对象
面试经常会问。
A2A 主要有四个核心对象：
| 对象 | 作用 |
|---|---|
| Agent Card | Agent能力描述 |
| Task | 任务 |
| Message | 消息 |
| Artifact | 执行结果 |

---

## Agent Card
描述 Agent。
类似：
```text
OpenAPI
Swagger
```
---
## Message
Agent 之间传递消息。
例如：
```text
任务请求
任务结果
状态更新
```
---
## Task
表示一个工作任务。
例如：
```text
生成报告
分析日志
代码审查
```
---
## Artifact
任务产出物。
例如：
```text
PDF
Word
文本
图片
分析结果
```

# A2A 和 MCP 的协作关系
很多面试官会追问。
关系如下：
```text
Agent
  │
  ├─ A2A → 调用其它Agent
  │
  └─ MCP → 调用工具
```
例如：
```text
HR Agent
      │
      ├─ A2A
      ▼
Finance Agent
Finance Agent
      │
      ├─ MCP
      ▼
MySQL
```

因此：

**A2A 管 Agent 协作。**
**MCP 管 Tool 调用。**

两者并不冲突。

# 面试回答
A2A 是 Google 推出的 Agent 间通信协议，其核心工作流程包括 Agent 发现、能力发现、任务创建、任务执行、状态同步以及结果返回。每个 Agent 会通过 Agent Card 暴露自身能力，调用方根据能力描述选择合适的 Agent 发起任务请求。任务执行过程中支持状态跟踪和异步通知，最终通过 Artifact 返回执行结果。
A2A 的本质是为多智能体系统建立统一通信标准，使不同框架、不同厂商开发的 Agent 能够像微服务一样协同工作，是未来多 Agent 系统的重要基础设施。

# 面试官追问

### 追问1：Agent Card 的作用是什么？
**答案：**
Agent Card 类似于 OpenAPI 或 Swagger 文档。
用于描述：
- Agent名称
- Agent能力
- 支持任务
- 调用方式
其它 Agent 可以根据 Agent Card 判断是否调用该 Agent。

---
### 追问2：为什么 A2A 支持异步任务？
**答案：**
因为很多 Agent 任务耗时较长。
例如：
- 深度研究
- 数据分析
- 代码生成
- 企业审批流
可能持续几分钟甚至几小时。
因此 A2A 提供任务状态同步机制，实现长时间任务管理。

---
### 追问3：A2A 和微服务有什么相似之处？
**答案：**
两者都强调：
- 服务自治
- 松耦合
- 能力暴露
- 网络通信
区别在于：
微服务暴露的是业务接口。
A2A 暴露的是 Agent 能力。
可以理解为：

```text
微服务时代：
Service → Service
AI时代：
Agent → Agent
```
