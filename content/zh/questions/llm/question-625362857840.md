## 答案

## 简短答案
RAG（Retrieval-Augmented Generation，检索增强生成）的完整流程可以概括为：
**知识库构建 → 文档切片 → Embedding 向量化 → 存储向量库 → 用户提问 → 问题向量化 → 混合检索 → Rerank 重排序 → Prompt 构建 → LLM 生成答案 → 返回结果**

企业级项目通常会进一步优化为：
**文档解析 → Chunk 切分 → Embedding → 向量存储 → Hybrid Search → Rerank → Prompt Engineering → LLM → Answer**

---
# RAG 为什么会出现？
传统大模型存在三个明显问题：
**第一，知识有时效性。**
例如：
```text
今天发生的新闻
昨天发布的新产品
企业最新制度
```
模型训练时并不知道。
**第二，无法掌握企业私有知识。**
例如：
```text
公司内部文档
运维手册
产品说明书
ERP流程
工单系统知识库
```
这些内容不会出现在公开训练数据中。
**第三，容易产生幻觉。**
模型不知道答案时，可能会根据概率强行生成一个看起来合理但实际上错误的答案。
因此出现了 RAG。
RAG 的核心思想非常简单：
> 先查资料，再回答问题。

---
# 第一阶段：知识库构建
这是离线阶段。
目的是把企业知识整理成可检索的数据。
知识来源通常包括：
- PDF
- Word
- Excel
- Markdown
- 数据库
- FAQ
- 网页内容
- 运维文档
例如：
```text
员工手册.pdf
Nacos运维手册.docx
RocketMQ最佳实践.md
MySQL设计规范.xlsx
```
这些文件首先需要进入知识库系统。

---
# 第二阶段：文档解析
系统需要读取文档内容。
例如：
```text
PDF
↓
提取文本
Word
↓
提取文本
Excel
↓
提取文本
```
最终统一变成纯文本。
例如：
```text
Nacos 是阿里开源的注册中心和配置中心。
```
---
# 第三阶段：文档切片（Chunk）
这是 RAG 非常关键的一步。
因为大模型无法一次读取几十万字的文档。
所以需要切分。
例如：
原始文档：
```text
SpringCloud教程
共100页
```
切片后：
```text
Chunk1
Nacos介绍
Chunk2
服务注册流程
Chunk3
配置中心原理
Chunk4
集群部署方案
```
常见切片大小：
```text
200~1000 Token
```
企业项目中通常：
```text
500 Token 左右
```
效果较好。

---
# 第四阶段：Embedding 向量化
Embedding 的作用是：
**把文本转换成机器可以计算的向量。**
例如：
```text
Nacos是什么？
```
转换后：
```text
[0.12,0.45,0.33,0.78...]
```
再例如：
```text
Nacos有什么作用？
```
转换后：
```text
[0.13,0.46,0.31,0.79...]
```
因为语义接近。
所以两个向量距离也接近。
常见模型：
- BGE-M3
- BGE-Large
- Qwen Embedding
- OpenAI Embedding
- Jina Embedding
---
# 第五阶段：存储向量数据库
向量生成后需要存储。
常见向量数据库：
- Milvus
- PGVector
- Elasticsearch Vector
- Chroma
- Weaviate
- FAISS
存储内容一般包括：
```json
{
  "content":"Nacos是注册中心",
  "vector":[0.12,0.55,0.77],
  "source":"nacos.pdf"
}
```
---
# 第六阶段：用户提问
例如用户输入：
```text
Nacos是干什么的？
```
此时开始进入在线检索阶段。

---
# 第七阶段：问题向量化
与知识库处理方式一样。
问题也要进行 Embedding。
例如：
```text
Nacos是干什么的？
```
转换：
```text
[0.15,0.53,0.82...]
```
---
# 第八阶段：混合检索（Hybrid Search）
这是现代 RAG 的核心优化之一。
通常会同时进行：
**关键词检索（BM25）**
和
**向量检索（Vector Search）**
例如：
用户提问：
```text
Nacos服务发现原理
```
关键词检索找到：
```text
包含Nacos
包含服务发现
```
的文档。
向量检索找到：
```text
服务注册机制
注册中心工作流程
```
等语义相近内容。
然后将结果合并。
形成：
```text
Top20候选文档
```
---
# 第九阶段：Rerank 重排序
向量检索只是粗召回。
不一定排序准确。
因此需要：
```text
Top20
↓
Rerank
↓
Top3
```
例如：
用户问题：
```text
Nacos如何实现服务注册？
```
召回结果：
```text
Nacos安装教程
Nacos服务注册机制
Nacos配置中心
```
Rerank 会重新计算相关性。
最终排序：
```text
1. Nacos服务注册机制
2. Nacos配置中心
3. Nacos安装教程
```
只保留最相关内容。
常见模型：
- BGE-Reranker
- Jina Reranker
- Cohere Rerank
---
# 第十阶段：Prompt 构建
将用户问题和检索结果拼接。
例如：
```text
你是一名Java专家。
参考资料：
Nacos是阿里开源的注册中心。
Nacos通过心跳机制维护服务实例。
用户问题：
Nacos如何实现服务注册？
```
形成最终 Prompt。

---
# 第十一阶段：LLM 生成答案
最终发送给：
- DeepSeek
- GPT
- Gemini
- Qwen
模型根据上下文生成答案。
例如：
```text
Nacos通过服务注册、心跳续约和服务发现机制实现微服务治理。
```
---
# 第十二阶段：返回结果
最终返回给用户。
很多企业系统还会增加：
- 引用来源
- 原文链接
- 相似问题
- 可信度评分
例如：
```text
答案
引用来源：
Nacos官方文档.pdf
可信度：
95%
```
---
# 企业级 RAG 完整架构
```text
离线阶段
文档采集
    ↓
文档解析
    ↓
Chunk切分
    ↓
Embedding
    ↓
向量数据库
在线阶段
用户问题
    ↓
Embedding
    ↓
Hybrid Search
(BM25 + Vector)
    ↓
TopN召回
    ↓
Rerank
    ↓
TopK文档
    ↓
Prompt构建
    ↓
LLM
    ↓
最终答案
```

# 面试回答

RAG 即检索增强生成，其核心思想是在大模型回答问题之前，先从外部知识库检索相关内容，再将检索结果作为上下文提供给大模型生成答案。
完整流程包括：文档采集、文档解析、Chunk 切分、Embedding 向量化、向量库存储、用户提问、问题向量化、混合检索、Rerank 重排序、Prompt 构建以及 LLM 生成答案。
在企业级项目中，通常采用 Hybrid Search + Rerank 的架构，其中混合检索负责提高召回率，Rerank 负责提高排序准确率，两者结合能够显著降低幻觉并提升回答质量。

# 面试官追问
### 追问1：为什么要进行 Chunk 切分？
**答案：**
因为大模型上下文长度有限，而且整篇文档检索粒度太粗。
切片后：
- 检索更精准
- Token 消耗更少
- 召回效果更好
通常每个 Chunk 控制在 300~800 Token。

---

### 追问2：为什么企业级 RAG 要增加 Rerank？
**答案：**
Embedding 检索属于粗召回。
它负责尽可能找到相关文档，但排序不一定准确。
Rerank 属于精排序。
负责从 TopN 文档中选出最相关的内容提供给大模型，从而提升最终答案质量。

---
### 追问3：为什么现在都推荐 Hybrid Search？
**答案：**
因为关键词检索和向量检索各有优缺点。
关键词检索擅长：
- 错误码
- 接口名
- TraceId
- 专业术语
向量检索擅长：
- 语义理解
- 同义词匹配
- 自然语言问答
混合检索能够同时利用两者优势，因此已经成为企业级 RAG 系统的主流方案。
