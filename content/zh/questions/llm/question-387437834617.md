## 答案

## 一句话概括
自查询（Self-Query）是指让大模型先分析用户问题，自动提取查询条件和元数据过滤条件，然后再去检索知识库，从而提高 RAG 检索的准确率。
简单来说：
> 用户提问题 → LLM理解问题 → 自动生成检索条件 → 向量检索 + 元数据过滤 → 返回更精准结果

---
# 为什么 RAG 需要自查询？
传统 RAG 存在一个核心问题：
**用户的问题中往往包含大量隐含条件，但向量检索无法很好理解这些条件。**
例如用户提问：

```text
帮我找一下 2025 年以后发布的 Spring AI 相关文章
```
知识库中有：
```text
标题：Spring AI 入门
发布时间：2024
标题：Spring AI Agent开发
发布时间：2025
标题：Spring AI MCP实践
发布时间：2026
```
如果直接向量检索：
```text
query = "Spring AI 相关文章"
```
可能会把 2024 的文章也召回，因为向量数据库只能理解：

"Spring AI","相关文章" 而无法理解："发布时间 > 2025"

这种结构化条件,所以需要 Self-Query。

---

# Self-Query 的核心思想

让 LLM 先充当一个查询分析器（Query Parser）。

例如,用户问题：

```text
查找2025年以后发布的Spring AI相关文章
```

LLM先解析：

```json
{
  "query": "Spring AI",
  "filter": {
    "publish_time": {
      "$gte": "2025-01-01"
    }
  }
}
```

然后执行：
```text
向量检索：
Spring AI
元数据过滤：
publish_time >= 2025
```

最终返回：
```text
Spring AI Agent开发（2025）
Spring AI MCP实践（2026）
```

过滤掉：
```text
Spring AI 入门（2024）
```

# Self-Query 的工作流程

## 第一步：用户提问

```text
帮我找一下 RocketMQ 相关的高级文章
```
---

## 第二步：LLM分析问题

识别出：

```text
关键词：
RocketMQ
等级：
高级
```

生成：
```json
{
  "query":"RocketMQ",
  "filter":{
    "level":"advanced"
  }
}
```
---

## 第三步：构造检索请求

向量检索：
```text
RocketMQ
```

过滤条件：

```text
level = advanced
```
---

## 第四步：知识库检索

数据库：
| 文档 | level |
|---|---|
| RocketMQ入门 | beginner |
| RocketMQ事务消息 | advanced |
| RocketMQ源码分析 | advanced |

返回：

```text
RocketMQ事务消息
RocketMQ源码分析
```
---

## 第五步：生成最终答案
将召回结果放入 Prompt：
```text
Context:
RocketMQ事务消息...
RocketMQ源码分析...
```
然后生成答案。

# Self-Query 和普通向量检索的区别
| 对比项 | 普通向量检索 | Self-Query |
|---|---|---|
| 理解用户意图 | 弱 | 强 |
| 支持时间过滤 | ❌ | ✅ |
| 支持作者过滤 | ❌ | ✅ |
| 支持标签过滤 | ❌ | ✅ |
| 支持权限过滤 | ❌ | ✅ |
| 召回准确率 | 一般 | 更高 |
| 实现复杂度 | 低 | 高 |

# Self-Query 在企业级 RAG 中的应用
你做的企业智能助手就非常适合。
例如知识库文档：
```json
{
  "title":"RocketMQ部署文档",
  "department":"研发部",
  "author":"张三",
  "create_time":"2026-01-01",
  "tag":"中间件"
}
```
用户提问：
```text
帮我找研发部关于RocketMQ的最新部署文档
```
Self-Query生成：
```json
{
  "query":"RocketMQ 部署",
  "filter":{
    "department":"研发部",
    "create_time":"最新"
  }
}
```
检索精度会远高于纯向量搜索。

```python
LLM
```

```python
retriever = SelfQueryRetriever.from_llm(
```

```text
query:
Spring AI
```

# 面试官可能追问
## Q1：Self-Query 和 Query Rewrite 有什么区别？
### 简短回答
Query Rewrite 改写问题。
Self-Query 提取过滤条件。
### 举例

用户：

```text
最近发布的 Spring AI Agent 文章
```
Query Rewrite：

```text
Spring AI Agent 开发实践
```
Self-Query：

```json
{
  "query":"Spring AI Agent",
  "filter":{
    "publish_time":"最近"
  }
}
```

Rewrite解决语义表达问题。
Self-Query解决结构化过滤问题。

---
## Q2：Self-Query 和 Metadata Filter 有什么区别？

### 简短回答
Metadata Filter 是人工写规则。
Self-Query 是 LLM 自动生成规则。

### 例子

人工：
```python
filter = {
  "author":"张三"
}
```
Self-Query：
用户输入：
```text
张三写的文章
```
LLM自动生成：
```python
filter = {
  "author":"张三"
}
```
---
## Q3：Self-Query 的缺点是什么？
### 答案
主要有三个问题：
#### 1. 增加一次 LLM 调用
流程：
```text
用户问题
    ↓
LLM解析条件
    ↓
向量检索
    ↓
生成答案
```
会增加延迟。

#### 2. 过滤条件可能生成错误
例如：
```text
最近文章
```
LLM可能理解成：
```text
最近7天
```
也可能理解成：
```text
最近30天
```
存在歧义。

#### 3. 依赖元数据质量
如果知识库没有：
```text
作者
时间
标签
部门
```
这些字段，
Self-Query无法发挥作用。

---
## Q4：Self-Query、Hybrid Search、Rerank 三者关系是什么？

### 面试高频答案
三者分别优化不同阶段：

```text
Self-Query
    ↓
优化查询条件
Hybrid Search
    ↓
优化召回率
Rerank
    ↓
优化排序精度
```

完整企业级流程：
```text
用户问题
    ↓
Self-Query
    ↓
Hybrid Search
(BM25 + Vector)
    ↓
Top50
    ↓
Rerank
    ↓
Top5
    ↓
LLM生成答案
```

这是目前企业级 RAG 最常见的检索增强方案。
