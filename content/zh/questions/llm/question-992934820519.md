## 答案

## 简短答案

在 RAG 系统中，数据清洗和预处理是影响检索效果最重要的环节之一。
很多时候知识库效果不好，并不是 Embedding 模型不行，也不是向量数据库有问题，而是因为知识库数据质量太差。
数据预处理的核心目标是：
**让知识变得结构化、语义完整、便于切分、便于检索。**
通常包括以下几个步骤：
1. 文档解析
2. 内容清洗
3. 去重处理
4. 格式标准化
5. 无效内容过滤
6. 文档切块（Chunk）
7. 元数据提取
8. 关键词增强
9. 向量化存储

---

# 为什么数据清洗如此重要？

很多人刚接触 RAG 时会认为：
```text
PDF
↓
Embedding
↓
向量库
↓
结束
```
实际上这样做效果往往很差。
例如一份企业文档：
```text
====================
XX系统运维手册
版本：V2.1
作者：张三
联系电话：138xxxxxxxx
更新时间：2025-01-01
====================
```

如果这些内容全部进入向量库：

```text
版本号
联系电话
页码
版权声明
```

都会参与检索。
最终导致：

```text
召回不准
噪声增多
Rerank效果下降
Token浪费
```

因此企业级 RAG 一定会做数据清洗。

---
# 第一步：文档解析
首先需要把各种格式统一转换成文本。
常见格式包括：
- PDF
- Word
- Excel
- PPT
- Markdown
- HTML
- 数据库记录
例如：
```text
运维手册.pdf
```
解析后：
```text
Nacos 是阿里开源的注册中心与配置中心。
```
这一步通常使用：
- Apache Tika
- PDFBox
- Unstructured
- MinerU
- PaddleOCR

---

# 第二步：去除无效内容
这是最容易被忽略的一步。
例如：
```text
第1页
第2页
Copyright 2025
仅供内部使用
Confidential
```
这些内容对于问答毫无意义。
应该删除。
清洗前：
```text
第15页
Redis缓存设计方案
版权所有XX公司
```
清洗后：
```text
Redis缓存设计方案
```
这样 Embedding 质量会明显提升。

---
# 第三步：去除重复内容
企业文档经常存在：
```text
页眉
页脚
水印
版权声明
```
例如：
```text
青岛某公司内部文档
```
每页都会出现。
如果不处理：
```text
100页文档
出现100次
```
Embedding 后会污染向量空间。
因此需要统一去重。

---
# 第四步：格式标准化
不同文档格式差异很大。
例如：
原文：
```text
Redis缓存
Redis穿透
Redis雪崩
```
或者：
```text
Redis缓存 Redis穿透 Redis雪崩
```
需要统一处理。
包括：
- 空格规范化
- 换行规范化
- 标点统一
- 编码统一
例如：
```text
Redis缓存 Redis穿透 Redis雪崩
```
转换：
```text
Redis缓存
Redis穿透
Redis雪崩
```
提高后续切块效果。

---
# 第五步：OCR 纠错
如果知识来源是扫描版 PDF。
例如：
```text
Spr1ngCloud
```
OCR 后可能出现：
```text
SpringCIoud
```
或者：
```text
Nac0s
```
这种错误会严重影响检索。
因此需要：
- OCR纠错
- 拼写纠错
- AI自动校正
很多企业知识库项目都会增加这一层。

---
# 第六步：结构化提取
这一阶段非常关键。
例如文档：
```text
第一章 Redis
1.1 Redis简介
1.2 Redis数据结构
第二章 缓存设计
```
应该保留层级关系。
例如：
```json
{
  "title":"Redis简介",
  "chapter":"第一章 Redis",
  "content":"Redis是..."
}
```

这样后续检索效果更好。

---
# 第七步：Chunk 切块
这是 RAG 效果影响最大的步骤之一。
错误切法：
```text
固定500字符
```
可能出现：
```text
第一段讲Redis
第二段讲MySQL
切块时切在中间
```
导致语义断裂。
正确做法：
按照：
- 标题
- 段落
- 章节
进行语义切分。
例如：
```text
Redis简介
↓↓↓
Chunk1
```
```text
Redis数据结构
↓↓↓
Chunk2
```
保持语义完整。

---
# 第八步：Chunk Overlap
很多知识跨 Chunk。
例如：
```text
Chunk1
Redis持久化包括RDB
```
```text
Chunk2
和AOF两种方式
```
信息被拆断。
因此需要重叠。
例如：
```text
Chunk1
0~500
```
```text
Chunk2
450~950
```
重叠：
```text
50 Token
```
常见设置：
```text
Overlap 10%~20%
```
---
# 第九步：元数据提取
除了正文。
还应保存：
```json
{
  "title":"Redis缓存设计",
  "author":"张三",
  "department":"研发中心",
  "source":"redis.pdf",
  "tag":"Redis"
}
```
检索时可以利用：
- Tag过滤
- 部门过滤
- 时间过滤
提高准确率。

---
# 第十步：关键词增强
很多企业知识存在同义词问题。
例如：
```text
服务发现
```
实际上：
```text
注册中心
```
也是同一个概念。
可以提前增加：
```text
服务发现
注册中心
Service Discovery
```
作为扩展关键词。
提升召回率。
这一步也叫：
**Query Expansion（查询扩展）**

---
# 企业级 RAG 数据处理流程
通常流程如下：
```text
文档采集
    ↓
文档解析
    ↓
OCR识别
    ↓
内容清洗
    ↓
去重处理
    ↓
格式标准化
    ↓
结构化提取
    ↓
Chunk切块
    ↓
Chunk Overlap
    ↓
元数据提取
    ↓
Embedding
    ↓
向量数据库
```
---
# 面试回答
在 RAG 系统中，数据清洗和预处理直接决定最终检索质量。通常需要先完成文档解析，将 PDF、Word、Excel 等内容统一转换为文本；随后去除页眉页脚、版权声明、水印等无效内容，并进行格式标准化和重复数据清理。对于扫描件还需要 OCR 识别和纠错。
之后会根据文档结构进行语义切块，并通过 Chunk Overlap 保证上下文连续性。同时提取标题、标签、来源等元数据，为后续过滤检索提供支持。最后再进行 Embedding 向量化并存入向量数据库。
在实际项目中，数据质量对 RAG 的影响往往超过模型本身，一个高质量知识库通常比更换 Embedding 模型带来的收益更大。

# 面试官可能的追问

### 追问1：为什么不能直接把 PDF 全量丢进向量库？

**答案：**
因为 PDF 中存在大量噪声数据：
- 页码
- 页眉页脚
- 版权声明
- 水印
- 联系方式
这些内容会污染向量空间，降低召回准确率，因此必须先清洗。
---
### 追问2：Chunk 为什么不能简单按字符数切分？
**答案：**
因为固定长度切分容易破坏语义完整性。
例如一段 Redis 内容被切成两半：
```text
Chunk1
Redis支持RDB
```
```text
Chunk2
和AOF持久化
```
这样检索时可能召回不到完整信息。
因此企业项目通常采用基于标题、段落、章节的语义切分。

---
### 追问3：Chunk Overlap 的作用是什么？
**答案：**
Overlap 用于保留相邻 Chunk 的上下文信息。
例如：
```text
Chunk1
0~500 Token
```
```text
Chunk2
450~950 Token
```
保留 50 Token 重叠区域。
这样即使重要内容位于边界位置，也不会因为切块导致语义丢失，从而提升检索召回率。
