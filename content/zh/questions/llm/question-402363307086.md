## 答案

## 简短答案
查询扩展（Query Expansion）是 RAG 检索优化中的一种常见技术。
它的核心思想是：
> 在用户原始问题的基础上，自动补充同义词、近义词、相关术语或改写后的问题，从而提高知识库召回率。

例如用户提问:

```text
Nacos 如何实现服务发现？
```
系统可能自动扩展为：
```text
Nacos 如何实现服务发现？
Nacos 注册中心原理是什么？
Nacos Service Discovery 如何实现？
Nacos 服务注册与发现机制是什么？
```
然后同时进行检索。
这样能够找到更多相关内容。

---
# 什么是查询扩展？
先看一个例子。
知识库中存储的是：
```text
Nacos 注册中心工作原理
```
用户提问：
```text
Nacos 服务发现是如何实现的？
```
对于人来说：
```text
服务发现
≈
注册中心
```
是同一个概念。
但是检索系统不一定知道。
因此可能出现：
```text
知识存在
却检索不到
```
的问题。
于是需要查询扩展。
系统自动把问题扩展为：
```text
服务发现
注册中心
Service Discovery
服务注册
```
然后一起检索。
这就是 Query Expansion。

---

# 为什么需要查询扩展？
RAG 最大的问题之一：
**用户的表达方式和知识库的表达方式不一致。**
例如：
知识库写的是：
```text
Redis缓存穿透
```
用户提问：
```text
缓存一直查数据库怎么办？
```
知识库写的是：
```text
JWT认证
```
用户提问：
```text
登录状态如何保持？
```
知识库写的是：
```text
Nacos注册中心
```
用户提问：
```text
服务发现机制
```
虽然本质上是同一个问题。
但是关键词完全不同。
如果直接检索：
```text
召回率下降
```
因此需要查询扩展。

---

# 查询扩展解决什么问题？
主要解决三个问题。
### 第一：同义词问题
例如：
```text
缓存穿透
```
可能被描述为：
```text
缓存失效
缓存没命中
缓存一直查数据库
```
如果只检索原词：
```text
容易漏召回
```
---
### 第二：专业术语问题
例如：
用户说：
```text
服务发现
```
知识库写：
```text
Service Discovery
```
或者：
```text
注册中心
```
扩展后：
```text
服务发现
注册中心
Service Discovery
```
召回率明显提升。

---
### 第三：用户表达不规范
例如：
用户提问：
```text
Redis为啥这么快？
```
扩展后：
```text
Redis性能为什么高？
Redis高性能原理是什么？
Redis为什么响应速度快？
```
这样更容易匹配知识库内容。

---

# 查询扩展有哪些实现方式？
## 第一种：同义词词典扩展
最简单的方法。
提前维护：
```json
{
  "服务发现":[
      "注册中心",
      "Service Discovery"
  ],
  "缓存穿透":[
      "缓存失效",
      "缓存没命中"
  ]
}
```
检索时自动扩展。
优点：
- 实现简单
- 性能高
缺点：
- 维护成本高
- 覆盖范围有限

---
## 第二种：LLM 改写（Rewrite）
利用大模型生成多个相似问题。
例如：
用户问题：
```text
Redis为什么快？
```
改写：
```text
Redis高性能原理是什么？
Redis响应速度快的原因有哪些？
Redis性能优势体现在哪些方面？
```
然后同时检索。
这种方式也是目前最主流的方案。

---
## 第三种：Multi Query Retrieval
也叫：
**多路查询检索**
流程：
```text
原问题
    ↓
LLM生成多个问题
    ↓
分别检索
    ↓
结果合并
```
例如：
```text
Redis为什么快？
```
生成：
```text
Redis性能高原因
Redis单线程优势
Redis内存数据库原理
Redis响应速度为什么快
```
然后：
```text
检索4次
```
最后合并结果。
LangChain、LangGraph、Spring AI 中都支持类似方案。

---
## 第四种：HyDE
全称：
```text
Hypothetical Document Embedding
```
假设文档检索。
流程：
```text
问题
 ↓
LLM先生成一个假答案
 ↓
对假答案Embedding
 ↓
检索知识库
```
例如：
问题：
```text
Nacos如何实现服务发现？
```
LLM先生成：
```text
Nacos通过注册中心维护服务实例...
```
然后用这段内容进行向量检索。
很多时候比直接检索效果更好。

---
# 查询扩展在 RAG 中的位置
完整流程：
```text
用户问题
      ↓
Query Rewrite
      ↓
Query Expansion
      ↓
Hybrid Search
      ↓
Rerank
      ↓
Prompt
      ↓
LLM
```
现代企业级 RAG 基本都会增加：
```text
Query Rewrite
+
Query Expansion
```
步骤。

知识库中有：

# Query Expansion 和 Rerank 的区别
很多面试官喜欢一起问。
| 技术 | 作用 |
|---|---|
| Query Expansion | 提高召回率 |
| Hybrid Search | 找更多候选文档 |
| Rerank | 提高排序准确率 |

流程：
```text
Query Expansion
        ↓
Hybrid Search
        ↓
Rerank
```
Query Expansion 解决：
```text
找不到
```
问题。
Rerank 解决：
```text
排序不准
```
问题。

# 面试回答
查询扩展（Query Expansion）是 RAG 检索优化技术之一，其核心思想是在用户原始问题基础上自动生成同义词、近义词、相关术语或多个改写版本，然后利用这些扩展查询共同检索知识库，提高召回率。
因为用户的表达方式与知识库中的表述往往存在差异，例如用户说“服务发现”，知识库写的是“注册中心”，如果直接检索可能无法召回相关内容。通过查询扩展，可以补充相关语义信息，减少漏召回问题。
实际项目中常见实现方式包括同义词词典扩展、LLM Query Rewrite、多路查询检索（Multi Query Retrieval）以及 HyDE 等技术。目前很多企业级 RAG 系统都会将 Query Expansion、Hybrid Search 和 Rerank 组合使用，以获得更好的检索效果。

# 面试官可能的追问

### 追问1：查询扩展和查询改写（Query Rewrite）有什么区别？

**答案：**

Query Rewrite 是：
```text
把一个问题改写成更标准的问题
```
例如：
```text
Redis为啥快
↓
Redis高性能原理是什么
```
Query Expansion 是：
```text
生成多个相关问题
```
例如：
```text
Redis为什么快
Redis高性能原理
Redis响应速度快原因
Redis性能优势
```
因此：
```text
Rewrite
解决表达不规范
Expansion
解决召回不足
```
---
### 追问2：查询扩展会不会带来副作用？
**答案：**
会。
如果扩展过度：
```text
引入无关问题
```
可能导致：
```text
召回噪声增加
```
例如：
```text
Redis为什么快
```
扩展出了：
```text
MySQL性能优化
```
显然不相关。
因此通常需要结合：
```text
Hybrid Search
Rerank
```
过滤无关结果。

---
### 追问3：企业项目里最常见的查询扩展方案是什么？

**答案：**
目前最常见的是：
```text
LLM Query Rewrite
+
Multi Query Retrieval
```
即利用大模型生成多个语义相近的问题，然后同时检索知识库，再结合 Rerank 排序。
这种方式比传统同义词词典覆盖范围更广，维护成本更低，也是当前主流的企业级 RAG 实践方案。
