## 答案

## 简短答案
Rerank（重排序）是 RAG 检索流程中的一个优化步骤。
它位于：
```text
向量检索之后
LLM生成之前
```
作用是：
> 对向量数据库召回的多个候选文档再次进行相关性评分，并重新排序，从而选出真正与用户问题最相关的内容提供给大模型。
Rerank 的核心目标是：
```text
提高召回结果质量
降低无关内容干扰
减少模型幻觉
提升最终答案准确率
```

# 什么是 Rerank？
先看一个例子。
用户提问：
```text
SpringCloud 中 Nacos 是如何实现服务注册与发现的？
```
向量检索返回：
```text
Top1：Nacos安装教程
Top2：Nacos配置中心介绍
Top3：Nacos服务注册原理
Top4：SpringBoot整合Nacos
Top5：Nacos控制台使用
```
虽然都与 Nacos 有关。
但是：
```text
真正最相关的是：
Nacos服务注册原理
```
而向量检索可能因为语义相近问题排序不准确。
因此需要：
```text
再次计算问题与文档的相关性
重新排序
```
变成：
```text
Top1：Nacos服务注册原理
Top2：SpringBoot整合Nacos
Top3：Nacos配置中心介绍
Top4：Nacos安装教程
Top5：Nacos控制台使用
```
这一步就叫：
```text
Rerank（重排序）
```

# 为什么需要 Rerank？
## 原因一：Embedding 检索不够精确
向量检索本质是：
```text
语义相似度匹配
```
例如：
用户问：
```text
Redis缓存穿透如何解决？
```
可能召回：
```text
Redis缓存穿透
Redis缓存雪崩
Redis缓存击穿
```
因为这些概念语义接近。
Embedding 很难区分细节。
---
## 原因二：向量数据库只负责粗召回
向量检索通常追求：
```text
高召回率
```
即：
```text
宁可多找
不要漏找
```
例如：
```text
先找20篇文档
```
但真正有价值的：
```text
可能只有前3篇
```
因此需要 Rerank 精排。
---
## 原因三：减少 Token 消耗
假设：
```text
召回20个Chunk
```
全部送给大模型：
```text
上下文巨大
成本增加
速度变慢
```
Rerank 后：
```text
只保留最相关的3~5个Chunk
```
即可大幅降低 Token 消耗。
---
# RAG 的完整流程
不带 Rerank：
```text
用户问题
    ↓
Embedding
    ↓
向量检索
    ↓
TopK文档
    ↓
LLM
```
带 Rerank：
```text
用户问题
    ↓
Embedding
    ↓
向量检索
    ↓
Top20文档召回
    ↓
Rerank重排序
    ↓
Top3文档
    ↓
LLM
```
企业级项目一般采用：
```text
召回 + 精排
```
两阶段检索方案。
---
# Rerank 的工作原理
## 第一阶段：Recall（召回）
向量检索：
```text
Question
      ↓
Embedding
      ↓
Milvus
      ↓
Top20
```
特点：
```text
速度快
准确率一般
```
---
## 第二阶段：Rerank（精排）
Rerank 模型逐个计算：
```text
问题
+
候选文档
```
之间的相关度。
例如：
```text
问题：
Nacos如何实现服务注册？
文档A：
Nacos安装教程
相关度：
0.45
```
```text
问题：
Nacos如何实现服务注册？
文档B：
Nacos服务注册流程
相关度：
0.97
```
最终排序：
```text
文档B
文档A
```
---
# Rerank 与 Embedding 的区别
| 对比项 | Embedding | Rerank |
|----------|----------|----------|
| 阶段 | 粗召回 | 精排序 |
| 输入 | 单独文本 | 问题+文档 |
| 输出 | 向量 | 相关度分数 |
| 速度 | 快 | 较慢 |
| 精度 | 一般 | 高 |
| 适合 | 大规模检索 | 小规模排序 |

# Spring AI 中的实现思路
伪代码：
```java
String question = "Nacos如何实现服务注册";
// 根据用户的问题 question，去向量数据库中查找最相似的知识片段，并返回。
List<Document> docs =
        vectorStore.similaritySearch(question);
//  Rerank（重排序）        
List<Document> rerankDocs =
        rerankModel.rerank(question, docs);
// 把问题 + 知识片段 拼成最终 Prompt
String prompt =
        buildPrompt(question, rerankDocs);
// Generation（生成答案）
chatModel.call(prompt);
```
流程：
```text
向量检索
      ↓
获取TopN
      ↓
Rerank排序
      ↓
取TopK
      ↓
生成答案
```

# 面试回答
Rerank 是 RAG 中的重排序技术，通常位于向量检索之后、大模型生成之前。
向量检索负责从海量知识库中快速召回候选文档，但召回结果的排序未必准确，因此需要使用 Rerank 模型再次计算用户问题与候选文档之间的相关性，并按照相关度重新排序。
典型流程是：
```text
用户问题
→ Embedding
→ 向量检索
→ Top20召回
→ Rerank排序
→ Top3保留
→ Prompt构建
→ LLM生成答案
```
Rerank 能够提高检索准确率、减少无关内容干扰、降低 Token 消耗，并显著提升最终回答质量。目前企业级 RAG 系统几乎都会引入 Rerank 作为标准配置。

# 面试官可能的追问
## 追问1：为什么向量检索之后还要 Rerank？
### 答案
因为向量检索属于：
```text
粗召回
```
目标是：
```text
不要漏掉相关文档
```
而不是：
```text
保证排序绝对正确
```
Rerank 属于：
```text
精排序
```
能够进一步提高相关性。

---
## 追问2：Rerank 会不会影响性能？
### 答案
会。
因为需要逐个计算：
```text
问题
+
候选文档
```
之间的相关度。
所以：
```text
Embedding速度快
Rerank速度慢
```
通常做法：
```text
先召回20条
再排序20条
最终保留3条
```
控制性能开销。

---
## 追问3：什么时候必须使用 Rerank？
### 答案
以下场景强烈建议使用：
```text
知识库内容很多
Chunk数量巨大
专业领域知识库
法律文档
医疗文档
企业内部知识库
```
如果知识库只有几十篇文档：
```text
Embedding检索即可
```
但如果达到：
```text
几千~几百万Chunk
```
通常必须增加 Rerank，否则检索准确率会明显下降。
