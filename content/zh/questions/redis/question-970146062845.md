## 答案

## 简要回答

`Redlock` 是 Redis 提出的多实例分布式锁算法：客户端只有在多个相互独立的 Redis 主节点中过半成功加锁，且耗时未超过锁有效期，才认为获得锁。它能降低单节点故障带来的风险，但不应被视为绝对强一致锁方案。

## 专业名词解释

### 分布式锁

分布式锁用于保证多个进程、服务实例或机器竞争同一资源时，同一时刻只有一个执行方进入关键业务逻辑。

### 租约

租约是带过期时间的锁。持有方只能在有效期内安全地认为自己拥有锁，超过有效期后，其他客户端可能重新获得该锁。

### Fencing Token

`Fencing Token` 是每次成功获得锁时生成的单调递增凭证。下游资源只接受更大的凭证，可拒绝因暂停、超时而恢复执行的旧持有者写入。

## 面试回答

`Redlock` 是 Redis 为提高单实例分布式锁容错能力提出的一种算法。它不是在一个 Redis 主从集群上做加锁，而是依赖多个相互独立的 Redis 主节点，官方算法通常以 5 个节点为例。

客户端加锁时，会使用相同的资源标识、唯一随机值和过期时间，依次向各节点执行类似 `SET key value NX PX ttl` 的操作，并为单次请求设置较短超时。只有当客户端在多数节点上成功加锁，例如 5 个节点中至少成功 3 个，并且整个加锁耗时小于锁的有效期时，才认为获取成功；否则需要释放已经加上的锁。释放锁时，必须校验随机值后再删除，避免误删其他客户端后续获得的锁。

`Redlock` 的价值在于避免单个 Redis 节点宕机就使锁不可用，适用于可接受少量异常风险的互斥控制场景。但它本质上仍依赖超时、时钟和客户端执行时序。当锁持有者发生长时间暂停、网络延迟或业务执行超过租约时间时，旧客户端仍可能继续操作共享资源。因此，对资金、库存扣减、主节点选举等正确性要求很高的场景，不能只依赖 `Redlock`；应结合 `Fencing Token`，或选择具备更强一致性语义的协调服务。

## 核心要点

- `Redlock` 是多 Redis 实例分布式锁算法
- 多数节点成功且未超时才算加锁成功
- 节点应相互独立，而不是主从副本
- 解锁必须校验唯一随机值再删除
- `Redlock` 不能天然解决过期后的旧请求
- 强一致场景应结合 fencing 或协调服务

## 面试追问

### 1. `Redlock` 和单 Redis 实例加锁有什么区别？

单实例锁通常通过 `SET key value NX PX ttl` 实现，逻辑简单、性能高，但 Redis 节点故障时会影响锁服务可用性。`Redlock` 将同一次加锁请求发送到多个独立节点，要求多数成功后才生效，从而降低单节点故障造成的影响，但部署和正确性分析也更复杂。

### 2. 为什么 `Redlock` 的多个 Redis 节点不能简单使用主从复制替代？

Redis 主从复制通常是异步的。若主节点加锁成功后尚未将锁同步到从节点就发生故障，从节点提升为新主后，另一个客户端可能再次成功加锁，导致两个客户端都认为自己持有锁。`Redlock` 要求的是相互独立的锁节点，而不是同一份锁数据的异步副本。

### 3. 为什么高一致性业务中仅使用 `Redlock` 仍可能不够安全？

即使客户端成功获得锁，也可能因长时间 GC、线程阻塞或网络暂停而超过锁有效期。此时新客户端已经取得锁，旧客户端恢复后仍可能继续写入。通过 `Fencing Token` 让下游拒绝旧编号请求，才能进一步防止过期持有者破坏数据正确性。

## 常见误区

**误区：** `Redlock` 使用多个 Redis 节点，因此可以保证任何情况下都不会出现并发执行。  
**正确理解：** `Redlock` 提升了部分故障场景下的容错能力，但仍受租约过期、暂停和网络时序影响，不能等价于绝对强一致互斥。

**误区：** Redis 主从集群中在主节点加一次锁，就等价于实现了 `Redlock`。  
**正确理解：** `Redlock` 依赖多个相互独立的 Redis 主节点进行多数派判断；异步主从切换可能丢失锁状态，不能直接替代该算法。
