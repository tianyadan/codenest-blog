## 答案

## 简要回答

Redis 复制延迟通常由**主节点写入量过大**、**网络带宽或延迟异常**、**从节点处理能力不足**、**慢命令**或**大 key 阻塞**，以及**全量同步**和**持久化**带来的资源竞争引起。

## 专业名词解释

### 复制延迟

复制延迟是指从节点已同步并处理的数据进度落后于主节点的现象。延迟过大时，从节点读取的数据可能不是最新状态。

### 全量同步

全量同步是指从节点无法继续基于已有复制进度追赶主节点时，需要重新接收完整数据集，再继续同步增量命令的过程。

## 面试回答

Redis 主从复制默认是异步的，因此从节点短暂落后于主节点是正常现象。复制延迟明显升高时，常见原因主要有五类。

第一，**主节点写入压力过高**。如果主节点持续产生大量写命令，从节点接收和执行变更的速度跟不上写入速度，复制偏移量差距就会扩大。

第二，**网络问题**。主从节点之间带宽不足、跨机房网络延迟高、丢包或短暂断连，都会影响复制数据传输；断连时间较长时，还可能导致无法部分同步，转而触发更重的全量同步。

第三，**从节点资源不足**。从节点 CPU、内存、网络或磁盘性能较弱，或者同时承担大量读请求，就可能无法及时处理复制流。

第四，**慢命令和大 key 操作**。主节点执行耗时命令会影响复制数据产生与发送，从节点执行大 key 相关命令也会拖慢追赶速度，例如读取或删除超大集合。

第五，**持久化或同步过程带来的抖动**。`RDB` 生成、`AOF` 重写、全量同步期间的数据传输与加载都可能消耗 CPU、内存、磁盘和网络资源，造成复制延迟升高。

排查时通常先查看 `INFO replication` 中主从连接状态、复制偏移量和延迟信息，再结合 `SLOWLOG GET`、延迟监控以及 CPU、内存、网络、磁盘指标定位根因。处理方式包括降低慢命令和大 key 影响、优化网络与实例资源、减少从节点额外读压力，并合理配置复制积压缓冲区以提高短暂断连后的部分同步成功率。

## 核心要点

- Redis 主从复制默认存在短暂延迟
- 高写入流量会加大主从进度差距
- 网络异常可能触发更重的全量同步
- 从节点负载过高会影响追赶速度
- 慢命令和大 key 会放大复制延迟
- 可通过 `INFO replication` 排查状态

## 面试追问

### 1. 如何判断 Redis 从节点是否存在复制延迟？

可以查看 `INFO replication` 输出中的主从连接状态、复制偏移量以及从节点延迟相关信息。若主从偏移量差距持续扩大，或从节点长期无法追上主节点，通常说明存在明显复制延迟，需要继续排查网络、写入量和实例负载。

### 2. 为什么网络闪断可能导致复制延迟突然升高？

网络短暂中断后，如果主节点的复制积压缓冲区仍保留从节点缺失的数据，从节点可以进行部分同步并快速追赶；如果缺失数据已经超出缓冲区范围，就需要重新进行全量同步，数据传输和加载成本更高，延迟也会明显增加。

## 常见误区

**误区：** 从节点已经连接成功，读取的数据就一定与主节点完全实时一致。  
**正确理解：** Redis 复制默认是异步的，从节点即使连接正常，也可能短暂落后于主节点，强一致读取场景不能直接依赖从节点。

**误区：** 复制延迟一定只是网络不稳定造成的。  
**正确理解：** 网络只是原因之一，主节点写入过高、从节点负载不足、慢命令、大 key、全量同步和持久化资源竞争都可能造成延迟。
