## 答案

## 简要回答

Redis 事务通过 `MULTI`、`EXEC`、`WATCH` 实现命令批量顺序执行和乐观锁控制；关系型数据库事务通常支持完整的 `ACID` 语义、执行失败回滚、隔离级别和复杂约束，两者能力并不等价。

## 专业名词解释

### ACID

`ACID` 通常指事务的**原子性**、**一致性**、**隔离性**和**持久性**。关系型数据库会围绕这四个目标提供较完整的事务机制。

### 乐观锁

乐观锁假设并发冲突较少，提交前再检测数据是否被修改。Redis 可通过 `WATCH` 监视 key，若提交前 key 已变化，则本次事务不执行。

### 回滚

回滚是指事务执行过程中失败后，撤销已经产生的数据修改，使数据恢复到事务开始前的状态。

## 面试回答

Redis 事务与关系型数据库事务最大的区别在于：Redis 更接近“将一组命令排队后连续执行”，而关系型数据库事务通常提供更完整的数据一致性保障。

Redis 使用 `MULTI` 开启事务，之后的命令不会立即执行，而是进入队列；调用 `EXEC` 后，队列中的命令会按顺序执行，执行期间不会穿插其他客户端的命令。也可以通过 `WATCH` 监听 key，在 `EXEC` 之前若 key 被其他客户端修改，则整个事务取消执行，这属于乐观锁机制。

```bash
WATCH account:1001
MULTI
DECRBY account:1001 100
INCRBY account:1002 100
EXEC
```

但 Redis 事务不支持关系型数据库那种运行期错误回滚。例如事务中第一条命令已经成功执行，后续命令因为类型错误而失败，Redis 不会撤销前面已经成功的修改，其余可执行命令仍会继续执行。只有在命令入队阶段就发现语法等错误时，`EXEC` 才会拒绝执行整个事务。

关系型数据库事务则通常可以在执行失败时通过 `ROLLBACK` 撤销已完成的修改，并支持读未提交、读已提交、可重复读、串行化等隔离级别，还可以结合唯一约束、外键约束和行锁处理复杂业务一致性。

因此，Redis 事务适合命令简单、逻辑可预校验、需要避免并发插入的原子操作；对于转账、库存结算等要求失败回滚和强一致性的核心业务，不能简单把 Redis 事务当成数据库事务替代品，通常应依赖数据库事务，或将逻辑封装为可控的 Lua 脚本并配合业务补偿。

## 核心要点

- Redis 事务基于 `MULTI` 和 `EXEC`
- `EXEC` 执行期间不会插入其他命令
- `WATCH` 用于实现乐观锁控制
- Redis 运行期错误不会自动回滚
- 数据库事务通常支持 `ROLLBACK`
- Redis 事务不能替代强一致业务事务

## 面试追问

### 1. Redis 事务中某条命令执行失败，会发生什么？

如果错误在入队阶段即可发现，例如命令语法错误，Redis 会在 `EXEC` 时拒绝执行该事务。如果命令已成功入队，但在执行阶段发生类型错误，失败命令会返回错误，其他命令仍会继续执行，已经成功的修改不会自动回滚。

### 2. WATCH 的作用是什么？它能实现数据库锁吗？

`WATCH` 用于监视一个或多个 key。在调用 `EXEC` 前，只要被监视的 key 发生变化，事务就不会执行，客户端通常需要重新读取数据并重试。它实现的是乐观并发控制，不是数据库中的悲观行锁，也不会阻止其他客户端修改数据。

### 3. Redis 中需要保证多步逻辑原子执行时，事务和 Lua 脚本如何选择？

`MULTI`/`EXEC` 适合客户端已经确定命令序列、只需批量连续执行的场景；若需要根据读取结果进行判断后再修改数据，Lua 脚本通常更合适，因为脚本内的读取、判断和写入可以在服务端一次原子执行，减少客户端往返与并发竞争。

## 常见误区

**误区：** Redis 事务与 MySQL 事务一样，任意命令失败都会自动回滚。  
**正确理解：** Redis 事务中执行阶段的命令错误不会触发整体回滚，之前成功执行的修改仍然保留。

**误区：** 使用 `WATCH` 后，其他客户端就不能再修改被监听的 key。  
**正确理解：** `WATCH` 不会锁住 key，只会在 key 被修改后使当前事务提交失败，属于乐观锁机制。
