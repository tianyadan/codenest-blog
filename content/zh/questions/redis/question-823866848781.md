## 答案

## 简要回答

`Ziplist` 是早期**用于紧凑存储少量小元素的连续内存结构**，节省内存但中间**修改成本较高**。`Quicklist` 是 Redis `List` 的主要底层结构，通过“**分段连续存储 + 双向链表**”在内存与操作性能之间取得平衡。

## 专业名词解释

### Ziplist

`Ziplist`，即压缩列表，是一块连续内存中依次编码多个字符串或整数元素的结构。它不为每个元素单独保存指针，因此非常节省内存。

### Quicklist

`Quicklist` 是由多个紧凑存储节点组成的双向链表。历史版本中每个节点内部通常是 `Ziplist`，Redis `7.0` 起相关内部存储改为 `Listpack`。

### 连锁更新

`Ziplist` 中每个元素需要记录前一个元素的长度。当某个元素变大后，可能导致后续元素的长度字段也需要扩展，从而连续修改多个元素，这种现象称为连锁更新。

## 面试回答

`Ziplist` 和 `Quicklist` 都是 Redis 为降低内存开销而设计的内部结构，但定位不同。

`Ziplist` 是一种**连续内存结构**，它将多个小字符串或整数紧凑地编码在同一块内存中，不像普通链表那样为每个节点保存前后指针，因此在元素少、内容短时非常节省内存。它支持从头尾访问元素，但因为数据连续存储，插入或删除可能需要移动后续数据；更典型的问题是元素长度变化时可能触发连锁更新，所以不适合承载很大的列表。

早期 Redis 的 `List` 在元素较少时可以使用 `Ziplist`，元素变多后转为普通双向链表。这样虽然兼顾了部分场景，但转换后的普通链表指针开销较大，内存利用率并不理想。

从 Redis `3.2` 开始，`List` 主要使用 `Quicklist`。它可以理解为“链表中的每个节点存一小段压缩数据”：外层是双向链表，方便从两端操作和节点级拆分；内层节点历史上保存一个 `Ziplist`，避免每个列表元素都承担链表指针开销。Redis 还可以压缩中间节点，仅保留两端节点便于快速访问，从而进一步节省长列表的内存。

需要注意版本变化：从 Redis `7.0` 开始，Redis 将原先使用 `Ziplist` 的相关编码迁移到 `Listpack`，因为 `Listpack` 延续了紧凑存储的优点，同时避免了 `Ziplist` 可能出现的连锁更新问题。因此面试中可以说：**`Ziplist` 是历史上的紧凑编码结构，`Quicklist` 是 `List` 的分段链表结构；新版本中 `Quicklist` 节点内部应理解为 `Listpack`，而不是继续强调 `Ziplist`。**

## 核心要点

- `Ziplist` 使用连续内存紧凑存储元素
- `Ziplist` 节省内存但修改成本较高
- `Ziplist` 可能发生连锁更新
- `Quicklist` 是分段紧凑存储的双向链表
- Redis `3.2` 起 `List` 主要使用 `Quicklist`
- Redis `7.0` 起相关节点内部改用 `Listpack`

## 面试追问

### 1. 为什么 Quicklist 比普通双向链表更节省内存？

普通双向链表通常需要为每个元素保存节点结构和前后指针，元素越多，额外开销越明显。`Quicklist` 将多个元素紧凑存放在一个内部节点中，外层只为一组元素维护链表指针，因此能显著减少节点与指针开销。

### 2. Ziplist 为什么可能出现连锁更新问题？

`Ziplist` 的元素会记录前一个元素的长度，便于反向遍历。当某个元素变大，使后续元素记录前置长度的字段不足时，后续元素也需要扩展；扩展又可能继续影响下一个元素，最终引发连续重分配和数据移动。

### 3. Redis 新版本中还能把 Quicklist 说成“链表加 Ziplist”吗？

只适合描述历史实现。Redis `3.2` 到较早版本中，`Quicklist` 节点内部确实与 `Ziplist` 相关；但 Redis `7.0` 已将相关紧凑编码切换为 `Listpack`。因此面试表达时应补充版本背景，避免把旧实现当作当前实现。

## 常见误区

**误区：** Ziplist 就是普通双向链表，只是节点更小。  
**正确理解：** Ziplist 本身是一块连续内存中的紧凑编码序列，并不是每个元素独立分配的链表节点。

**误区：** 当前所有 Redis 版本的 `Quicklist` 节点内部都使用 `Ziplist`。  
**正确理解：** `Quicklist` 历史上使用过 `Ziplist` 作为内部节点载体，但 Redis `7.0` 起相关实现已经迁移到 `Listpack`。
