## 答案

## 简要回答

Redis 可以通过 `List` 实现队列和栈：队列在一端入队、另一端出队，栈在同一端压入和弹出。队列常用 `LPUSH` + `RPOP`，栈常用 `LPUSH` + `LPOP`。

## 专业名词解释

### FIFO

FIFO 即先进先出，先进入队列的元素先被取出，是普通队列的基本规则。

### LIFO

LIFO 即后进先出，最后压入的元素最先被弹出，是栈的基本规则。

### 阻塞消费

阻塞消费是指列表为空时，消费者等待新元素到达，而不是不断轮询。Redis 中可通过 `BLPOP`、`BRPOP` 或 `BLMOVE` 实现。

## 面试回答

Redis 的 `List` 支持从左右两端进行插入和弹出操作，因此可以很方便地实现队列和栈。它的头尾写入、弹出操作适合高频数据流转场景。

实现**队列**时，需要遵循先进先出原则。例如使用 `LPUSH` 从左侧入队，再使用 `RPOP` 从右侧出队，最早进入的元素会最先被取出：

```bash
LPUSH task:queue task:1001
LPUSH task:queue task:1002

RPOP task:queue
RPOP task:queue
```

如果消费者不希望列表为空时反复轮询，可以使用阻塞式弹出命令，例如 `BRPOP task:queue 30`，在指定时间内等待任务到达。

实现**栈**时，需要遵循后进先出原则。可以使用 `LPUSH` 将元素压入左侧，再使用 `LPOP` 从同一侧弹出，最后压入的元素会最先被取出：

```bash
LPUSH page:stack page:A
LPUSH page:stack page:B

LPOP page:stack
LPOP page:stack
```

在简单任务队列中，直接使用 `RPOP` 或 `BRPOP` 有一个风险：任务一旦被弹出，就已经从队列移除，如果消费者处理过程中宕机，任务可能丢失。更稳妥的方式是使用 `LMOVE` 或 `BLMOVE`，将任务从待处理列表原子移动到处理中列表，处理成功后再删除确认；失败或超时的任务可以重新投递。

因此，`List` 很适合实现简单 FIFO 队列、栈、最近操作记录等功能；但如果业务需要消费组、消息确认、积压回溯和更完整的消息治理能力，更适合选择 Redis Streams 或专业消息队列。

## 核心要点

- 队列遵循先进先出原则
- 栈遵循后进先出原则
- 队列可用 `LPUSH` 配合 `RPOP`
- 栈可用 `LPUSH` 配合 `LPOP`
- 阻塞消费可使用 `BLPOP`、`BRPOP`
- 可靠任务处理可结合 `LMOVE`

## 面试追问

### 1. 实现队列时，能否使用 RPUSH 和 LPOP ？

可以。队列的关键是一端写入、另一端弹出，只要保证先进先出即可。`LPUSH` + `RPOP` 和 `RPUSH` + `LPOP` 都能实现 FIFO 队列，项目中应统一约定方向，避免生产者和消费者理解不一致。

### 2. BRPOP 和 RPOP 在队列消费中有什么区别？

`RPOP` 在队列为空时会立即返回空结果，消费者通常需要自行轮询；`BRPOP` 在队列为空时会等待元素到达或等待超时，可减少无效请求和 CPU 消耗，更适合简单的任务消费场景。

### 3. 如何降低消费者宕机导致的任务丢失风险？

不能简单地弹出任务后直接处理，因为弹出后任务已从待处理队列移除。可以通过 `LMOVE` 或 `BLMOVE` 将任务原子移动到处理中列表，业务处理完成后再确认删除；若处理超时，可扫描处理中列表并执行重试或重新入队。

## 常见误区

**误区：** Redis `List` 实现队列时，必须固定使用 `LPUSH` 和 `RPOP`。  
**正确理解：** 只要从一端入队、从另一端出队，就能满足先进先出；`RPUSH` 与 `LPOP` 同样可以实现队列。

**误区：** 使用 `BLPOP` 或 `BRPOP` 后，任务消费天然不会丢失。  
**正确理解：** 阻塞弹出只解决等待消费问题，元素弹出后消费者若异常退出，任务仍可能丢失；可靠处理需要处理中列表、确认和重试机制。
