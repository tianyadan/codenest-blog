## 答案

## 简要回答

Redis 通常通过 `SET lock_key unique_value NX PX expire_time` 原子加锁，并使用 Lua 脚本校验持有者后释放锁。关键是唯一标识、过期时间、原子解锁，以及业务超时时的续期处理。

## 专业名词解释

### 分布式锁

分布式锁用于协调多个进程或多个服务实例对同一共享资源的互斥访问，保证同一时刻只有一个执行者进入关键业务逻辑。

### 锁续期

锁续期是指业务尚未执行完成时，持锁方定期延长锁的过期时间，避免锁提前失效后其他线程进入临界区。

## 面试回答

Redis 实现分布式锁，核心是利用命令执行的**原子性**。加锁时使用 `SET lock_key unique_value NX PX expire_time`：`NX` 保证只有锁不存在时才能写入，`PX` 设置过期时间，防止客户端宕机后锁一直无法释放；`unique_value` 一般使用请求标识或 UUID，用于区分锁的真正持有者。

业务执行完成后，不能直接调用 `DEL` 释放锁，因为锁可能已经过期并被其他线程重新获取。此时原线程再删除，就会误删别人的锁。**正确做法是通过 Lua 脚本原子地判断锁值是否等于自己的唯一标识，匹配后再删除**。

如果业务执行时间可能超过锁过期时间，应合理设置超时时间，或使用带自动续期能力的方案，例如 `Redisson` 的看门狗机制。对于一致性要求极高、不能容忍锁失效后并发执行的业务，还需要结合业务幂等、状态校验或数据库约束，不能只依赖 Redis 锁本身。

```bash
SET order:lock:1001 request-uuid NX PX 30000
```

```java
String script = """
    if redis.call('get', KEYS[1]) == ARGV[1] then
        return redis.call('del', KEYS[1])
    else
        return 0
    end
    """;
```

## 核心要点

- 加锁应使用原子命令 `SET NX PX`
- 锁值必须保存唯一持有者标识
- 过期时间用于避免客户端宕机死锁
- 解锁必须校验锁值并原子删除
- 长耗时任务需要考虑锁续期
- 高一致性业务还需补充幂等校验

## 面试追问

### 1. 为什么不能使用 `SETNX` 后再单独执行 `EXPIRE`？

因为这两条命令不是原子操作。如果执行 `SETNX` 成功后，客户端在执行 `EXPIRE` 前宕机，锁就没有过期时间，可能长期无法释放。使用 `SET key value NX PX timeout` 可以一次性完成加锁和设置过期时间。

### 2. 为什么释放锁时不能直接执行 `DEL key`？

因为业务执行期间锁可能已经过期，随后被其他线程重新获取。原线程执行结束后直接 `DEL`，会把新持有者的锁删除，导致多个线程同时进入临界区。因此需要比较唯一标识后再删除，并使用 Lua 保证原子性。

### 3. 锁过期了，但业务还没有执行完成，应该怎么办？

可以根据业务耗时设置合理的锁过期时间，或者使用 `Redisson` 等具备自动续期机制的实现。续期时必须确认当前线程仍然持有该锁；同时业务本身应尽量具备幂等能力，避免极端情况下重复执行造成错误。

## 常见误区

**误区：** 只要执行 `SETNX` 成功，就能可靠实现分布式锁。  
**正确理解：** 可靠加锁还必须设置过期时间，并保证加锁与设置超时是原子操作，否则客户端异常时可能产生死锁。

**误区：** Redis 锁能够绝对保证业务永远不会重复执行。  
**正确理解：** 锁可能因超时、网络抖动或故障切换产生边界问题。重要业务仍应配合幂等设计、状态校验或数据库唯一约束。
