## 答案

**Redis 持久化主要有两种：**
1. RDB（Redis DataBase）
2. AOF（Append Only File）

Redis4.0 后：**还支持 RDB + AOF 混合持久化**。

**RDB**：类似“快照”，定期把当前 Redis 内存数据保存到磁盘 。
优点：恢复快、文件小。
缺点：可能丢失最后一次快照后的数据。

**AOF**：类似“操作日志”，Redis 每执行一条写命令，就追加记录到 AOF 文件。
优点：数据更安全。
缺点：文件更大、恢复更慢。

生产环境一般：AOF + RDB 混合使用。

**扩展知识：**

**一、为什么 Redis 需要持久化？**

Redis 的本质是内存数据库，数据都在内存里，如果Redis 宕机，或者服务器断电，内存数据会全部丢失。
所以 Redis 必须把数据保存到磁盘，这就是持久化。


**二、RDB 持久化（快照）**

RDB：
本质：就是某一时刻 Redis 内存数据的完整备份。类似“拍照片”。
比如Redis 当前：有100 万个 key，Redis 会生成 dump.rdb 文件，里面保存当前完整数据。


**三、RDB 如何触发？**

1 ） save（同步阻塞），生产一般不用， Redis 会阻塞当前线程，直到持久化完成，这是很危险的操作。
2 ） bgsave （后台异步快照），生产常用，Redis fork 子进程 ，子进程负责写 RDB 文件，主线程继续处理请求。
3 ） 自动触发 ：Redis 配置： save 900 1 、save 300 10 、save 60 10000

含义：
* 900 秒内有 1 次修改
* 或 300 秒内有 10 次修改
* 或 60 秒内有 10000 次修改
就触发 RDB。


**四、RDB 原理（重点）**

Redis：使用 fork 创建子进程。 子进程把当前内存数据写入磁盘。
这里利用Copy-On-Write（写时复制） 机制：
父进程：继续处理请求。
只有数据修改时才会复制对应内存页。

**五、RDB 优缺点：**

优点：
1 ） 文件小：压缩后的二进制文件。
2 ） 恢复速度快 ：直接加载整个快照。
3 ） 对性能影响较小 ：因为 bgsave 是子进程执行。
4 ） 适合备份，比如每天备份一次。

RDB 缺点
最大问题可能丢数据，例如：5 分钟做一次 RDB。如果Redis 刚写完数据 ，突然宕机那么最近 5 分钟数据全没了。

**六、AOF 持久化（追加日志）**
AOF：本质记录 Redis 写命令 ，例如  SET name tian HSET user 1 Tom  INCR count ， Redis 会把这些命令追加到：appendonly.aof 文件。


**七、AOF 恢复原理**
Redis 重启：重新执行 AOF 里的命令，从而恢复数据。类似：MySQL Binlog。


**八、AOF 刷盘策略（高频）**

1 ） always ：每条命令立刻刷盘，最安全，性能最差。
2 ） everysec（默认推荐）：每秒刷盘一次 ，性能与安全平衡。最多丢1 秒数据，生产最常用。
3 ） no ：由操作系统决定，性能高，但不安全。

**九、混合持久化（Redis4.0）**

Redis4 后支持：RDB + AOF 混合。
AOF Rewrite 时，前半部分：RDB 快照。后半部分：增量 AOF。
优点：恢复速度快，数据更安全。 这也是现在生产推荐方案。
