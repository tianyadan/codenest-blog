## 答案

## 简要回答

Redis 内存碎片化是指数据释放或变更后，已向操作系统申请的内存未能被高效复用或归还，**导致实际驻留内存明显高于有效数据内存**。优化时应先确认碎片来源，再考虑开启主动碎片整理、优化数据模型、平滑删除与必要时重启重载。

## 专业名词解释

### 内存碎片化

内存碎片化是指内存中存在大量不连续、难以复用或暂未归还给操作系统的空闲空间。Redis 看起来数据占用不高，但进程实际占用内存可能仍然较大。

### RSS

`RSS` 表示 Redis 进程实际驻留在物理内存中的大小。它与 Redis 统计的数据逻辑占用不同，通常可用于观察进程真实内存压力。

### 主动碎片整理

主动碎片整理是 Redis 在运行期间增量搬迁数据、回收碎片空间的机制，目的是降低内存浪费，同时尽量避免长时间阻塞业务请求。

## 面试回答

Redis 内存碎片化通常表现为：业务数据已经删除或过期，但进程占用的物理内存没有同步下降，或者 `INFO memory` 中的内存碎片相关指标持续偏高。原因是 Redis 频繁执行 key 的新增、删除、过期和 value 扩缩容后，内存分配器管理的空间可能形成零散空洞，虽然部分空间已经空闲，但不一定能立即复用或归还操作系统。

排查时，先查看 `INFO memory`，重点关注 `used_memory`、`used_memory_rss`、`mem_fragmentation_ratio` 以及分配器相关指标。需要注意，`mem_fragmentation_ratio` 偏高不一定全部来自可整理的分配器碎片，也可能包含进程自身开销、复制缓冲区或系统层面的影响，因此不能只凭一个比值直接下结论。

常见优化方式有四类。第一，**开启 Redis 主动碎片整理**，例如配置 `activedefrag yes`，让 Redis 在碎片达到阈值后增量整理内存；但整理过程会消耗 CPU，应结合延迟和负载观察效果。第二，**优化数据模型**，避免频繁创建和删除大量尺寸差异很大的 key，控制 big key，合理使用紧凑编码结构，并统一 value 大小特征。第三，**删除大 key 时使用 `UNLINK` 替代同步 `DEL`**，将释放工作异步化，减少主线程阻塞；同时合理设置过期时间，避免大量 key 在同一时间集中释放。第四，在允许维护窗口且碎片长期无法回落时，可以通过**从节点重建**、**主从切换**或**重启实例**重新装载数据，获得更紧凑的内存布局。

```bash
INFO memory

CONFIG SET activedefrag yes

UNLINK cache:large:key
```

此外，部分场景可尝试 `MEMORY PURGE` 请求内存分配器释放可归还的空闲页，但其实际效果依赖使用的分配器和当前内存布局，不能替代数据结构治理与主动碎片整理。

## 核心要点

- 碎片化会导致实际内存高于有效数据内存
- 频繁增删和 value 扩缩容容易产生碎片
- 排查应结合 `INFO memory` 多项指标
- `activedefrag` 可增量整理碎片空间
- `UNLINK` 可降低大 key 删除阻塞
- 严重碎片可通过重载实例彻底整理

## 面试追问

### 1. mem_fragmentation_ratio 越高，是否一定说明 Redis 存在严重碎片？

不一定。该指标反映进程驻留内存与 Redis 已使用内存之间的比例差异，除了分配器碎片，还可能包含进程额外开销、缓冲区和操作系统内存管理影响。因此需要结合 `allocator_frag_ratio`、业务数据变化、延迟和系统内存指标综合判断。

### 2. 开启 activedefrag 会带来什么影响？

`activedefrag` 会在运行期间主动搬迁对象、整理内存，从而降低碎片率并减少内存浪费。但整理本身需要消耗 CPU，负载较高时可能影响请求延迟。因此应在压测或监控基础上配置整理阈值和 CPU 使用范围，而不是无条件开启后忽略观察。

### 3. 删除大量 key 后内存没有立即下降，应该如何处理？

首先确认这些 key 是否真正删除，以及剩余内存是否来自碎片、缓冲区或持久化相关开销。对于大 key 删除，应优先使用 `UNLINK` 降低阻塞；碎片明显时可开启主动整理或尝试 `MEMORY PURGE`；若长期无法恢复且内存压力较大，可通过副本重建和切换方式整理内存布局。

## 常见误区

**误区：** 执行 `DEL` 删除大量 key 后，Redis 进程占用内存就一定会立即等比例下降。  
**正确理解：** key 被删除后，内存可能先归还给内存分配器而非立即归还操作系统，且零散空闲空间可能形成碎片，因此 `RSS` 不一定马上下降。

**误区：** 内存碎片化只能通过重启 Redis 解决。  
**正确理解：** Redis 可以通过主动碎片整理、优化 key 生命周期与数据结构、异步删除大 key 等方式缓解碎片；重启或重载通常是碎片严重且允许维护时的兜底方案。
