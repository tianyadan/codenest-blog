## 答案

在分布式锁（尤其是基于 Redis 的分布式锁）的实现中，EX 和 PX 都是用来**给锁设置过期时间的参数**，它们最核心的区别在于时间单位和精度不同：

-  EX (Expire in Seconds)：以**秒**为单位设置过期时间。
-  PX (Expire in Milliseconds)：以**毫秒**为单位设置过期时间。

可以参考下面的对比：

| 参数 |时间单位 | 精度| 示例 |
| --- | ---| --- | ---  |
| EX | 秒 (s) |较低，适合常规业务 |SET lock_key value NX EX 10 (10秒后过期) |
| PX | 毫秒 (ms) | 较高，适合精准控制 | SET lock_key value NX PX 10000 (10000毫秒后过期) |


**在分布式锁中的实际作用与建议：**

1 )  防止死锁：无论是 EX 还是 PX，在分布式锁中配合 NX（仅当键不存在时设置）使用，**核心目的都是为了防止死锁**。如果持有锁的客户端因为宕机或网络故障没有主动释放锁，Redis 会在设定的过期时间到达后自动删除这个键，从而让其他客户端有机会获取到锁。

2 )  如何选择：
- 在绝大多数常规的业务场景中，**使用 EX（秒） 已经完全足**够，且更加直观易读。
- 如果你的业务**对时间精度要求极高**（例如需要精确控制在几百毫秒内释放锁的高并发场景），或者锁的持有时间非常短，**那么建议使用 PX（毫秒）** 来进行更细粒度的控制。

**提醒：**

在设置分布式锁时，除了注意 EX 和 PX 的区别，**务必保证加锁操作是原子性的**（即使用一条 SET key value NX EX/PX time 命令），千万不要将加锁（SETNX）和设置过期时间（EXPIRE）拆分成两条命令执行，**否则在极端情况下依然可能导致死锁**。
