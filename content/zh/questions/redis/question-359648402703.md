## 答案

## 简要回答

Redis 通常使用有序集合 `ZSet` 实现排行榜：成员保存用户标识，`score` 保存积分或热度，Redis 会按分值自动排序，并支持快速查询排名、分数和榜单区间。

## 专业名词解释

### ZSet

`ZSet` 是 Redis 的有序集合类型。每个成员唯一，并关联一个可排序的 `score`，适合保存“用户 + 分数”这类排行榜数据。

### score

`score` 是 `ZSet` 中用于排序的数值，例如积分、点赞数或消费金额。分值相同时，Redis 会再按成员值的字典序排列。

## 面试回答

使用 Redis 实现排行榜，最常见的方案是使用 `ZSet`。可以把用户 ID 作为成员，把积分、热度或成绩作为 `score`。当用户得分变化时，通过 `ZINCRBY` 增加分值，或者通过 `ZADD` 直接更新分值，Redis 会自动维护排序结果。

查询排行榜时，如果分数越高排名越靠前，就使用 `ZREVRANGE` 获取前 N 名；查询某个用户当前排名可以使用 `ZREVRANK`，查询分数可以使用 `ZSCORE`。由于排名从 0 开始，展示给用户时通常需要再加 1。

实际项目中还要考虑排行榜周期和并列规则。例如日榜、周榜可以按日期生成不同的 key，如 `rank:score:2026-05-27`；如果存在同分用户，需要提前定义是允许并列展示，还是按更新时间、达成时间等规则进一步排序。对于历史榜单，可以设置过期时间或定期归档，避免 Redis 中保留过多无用数据。

```bash
ZADD rank:weekly 1001 user:1
ZINCRBY rank:weekly 20 user:1
ZREVRANGE rank:weekly 0 9 WITHSCORES
ZREVRANK rank:weekly user:1
ZSCORE rank:weekly user:1
```

## 核心要点

- 排行榜通常使用 Redis `ZSet`
- 用户标识作为成员，积分作为 `score`
- `ZINCRBY` 适合累加用户积分
- `ZREVRANGE` 可查询高分榜单
- `ZREVRANK` 查询排名时从 0 开始
- 周榜、日榜应使用不同业务 key

## 面试追问

### 1. 为什么排行榜适合使用 ZSet，而不是普通 Set 或 Hash ？

普通 `Set` 只能保证成员唯一，不能根据积分排序；`Hash` 适合存储字段和值，但无法直接维护排名。`ZSet` 同时支持成员唯一、按 `score` 排序、范围查询和排名查询，因此非常适合排行榜场景。

### 2. 如何查询某个用户的排名和前后几名用户？

可以先通过 `ZREVRANK` 获取用户排名，再根据该排名计算区间，使用 `ZREVRANGE` 查询附近成员。例如用户排名为 20，可以查询 18 到 22 名的数据，并在应用层处理越界情况。

### 3. 如何设计日榜、周榜和总榜？

可以分别使用不同 key，例如 `rank:daily:2026-05-27`、`rank:weekly:2026-W22` 和 `rank:total`。写入积分时根据业务需要同时更新对应榜单，短周期榜单设置合理过期时间，总榜长期保留或定期备份。

## 常见误区

**误区：** `ZSet` 会自动把相同分数的用户展示为相同名次。  
**正确理解：** `ZSet` 会对同分成员继续排序，`ZREVRANK` 返回的仍然是不同位置。是否展示为并列名次，需要业务层自行处理。

**误区：** 查询排名后直接展示 `ZREVRANK` 的返回值即可。  
**正确理解：** `ZREVRANK` 返回的排名下标从 0 开始，用户界面通常需要转换为从 1 开始的名次。
