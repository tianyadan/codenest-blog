## 答案


Redis 热点 Key 问题指的是：**某一个 key 在短时间内被大量高并发访问**。



**这会导致：**
* Redis 单节点压力暴增
* CPU 飙升
* 网络带宽打满
* 请求延迟增加
* 甚至缓存击穿数据库

**常见解决方案包括：**
1. 本地缓存
2. 多级缓存
3. 热点 key 拆分
4. Redis 集群分散压力
5. 限流与降级
6. 热点数据预加载
7. 永不过期 + 后台异步刷新
8. 使用 CDN / Nginx 缓存

**核心思想是： 不要让所有流量全部打到同一个 Redis key 上 。**


**扩展知识：**


一、什么是热点 Key？
热点 Key：就是某个 key 被疯狂访问。
比如：某个爆款商品：product:detail:1001 ，突然每秒几十万请求。所有请求全部访问这一个 key，这就是热点 Key。

二、热点 Key 会有什么问题？
1. Redis CPU 飙升： 因为Redis 单线程处理命令，大量请求打到一个 key，Redis 会忙不过来。
2. 网络带宽被打满：热点 key 的 value 如果还很大网络 IO 会非常夸张。
3. Redis 延迟变高 ：其他正常请求也会被拖慢。
4. 缓存击穿数据库：最危险的是热点 key 突然过期，然后大量请求同时查数据库，数据库直接崩。这叫缓存击穿

三、热点 Key 常见场景？
1. 秒杀商品
2. 热门文章
3. 热搜排行榜
4. AI 热门会话
5. 登录验证码
6. 热门视频信息
7. 首页推荐数据

四、如何解决热点 Key？

核心思想：分散压力。
方案一：本地缓存（最常用）
这是互联网大厂最经典方案。比如：Redis 前面再加一层 JVM 本地缓存 ：Caffeine 、Guava Cache
流程：请求 → 本地缓存 → Redis → DB

方案二：多级缓存
例如：浏览器缓存 → CDN → Nginx 缓存 → JVM 本地缓存 → Redis → MySQL 层层肖峰

方案三：热点 Key 拆分（重要）
例如：product:1001，所有请求都打这个 key。
可以拆成：
product:1001:1
product:1001:2
product:1001:3
product:1001:4

随机读。
这样：一个热点 key变成多个普通 key，压力被分散。
