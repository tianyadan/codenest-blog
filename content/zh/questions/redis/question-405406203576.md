## 答案

## 简要回答

Redis 的 `Zset` 是成员唯一、按 `score` 排序的集合。小数据量时通常使用 `Listpack` 紧凑存储；数据增大后使用**哈希表 + 跳表**，兼顾按成员查分和按分数排序查询的效率。

## 专业名词解释

### 跳表

跳表是在有序链表上增加多级索引的结构，通过逐层跳跃减少遍历范围，使插入、删除和查找的平均时间复杂度达到 `O(log N)`。

### Listpack

`Listpack` 是 Redis 的紧凑连续内存编码，适合保存少量、较短的成员与分值，能够减少指针和对象带来的额外内存开销。

## 面试回答

Redis 的 `Zset`，也叫有序集合，保存的是唯一的 `member` 与对应的浮点数 `score`。成员不能重复，但分数可以相同；排序时先按 `score` 升序排列，分数相同时再按 `member` 的字典序排列，因此它非常适合排行榜、延迟任务和滑动窗口限流等场景。

`Zset` 会根据数据规模选择不同的内部编码。在 Redis `7.0` 及之后版本中，当元素数量较少、成员长度较小时，通常使用 `Listpack`：成员和分值紧凑存放在连续内存中，内存开销较小，但查找、插入和范围定位通常需要遍历或移动数据，更适合小集合。更早版本中的同类紧凑编码是 `Ziplist`。

当集合超过紧凑编码阈值后，`Zset` 会转换为**哈希表 + 跳表**的组合结构。哈希表以 `member` 为 key，保存对应的 `score`，用于快速判断成员是否存在以及查询分值；跳表则按 `score` 和 `member` 顺序组织所有元素，用于高效实现排名、分数范围查询和顺序遍历。

之所以同时使用两种结构，是因为单独使用一种结构很难同时兼顾两类需求：只有哈希表，无法快速按分数排序和范围扫描；只有跳表，按成员直接查询分值不如哈希表高效。两者组合后，`ZADD` 更新分数时需要同步维护哈希表与跳表，典型的成员新增、删除、更新以及按排名定位都能保持较好的性能。

常见命令中，`ZADD` 用于新增成员或更新分数，`ZSCORE` 查询成员分值，`ZRANK` 或 `ZREVRANK` 查询排名，`ZRANGE` 可按排名或分数范围读取元素。需要注意，`Zset` 为排序能力付出了额外内存成本；如果业务只需要去重而不需要排序，应优先考虑 `Set`。

## 核心要点

- `Zset` 的成员唯一，分数可重复
- 相同分数按成员字典序排序
- 小集合使用 `Listpack` 节省内存
- 大集合采用哈希表与跳表组合
- 哈希表便于按成员快速查分
- 跳表便于排名和范围查询

## 面试追问

### 1. Redis 为什么使用跳表，而不是只用哈希表实现 Zset？

哈希表适合根据成员快速查询分数，但无法维护按分数排列的顺序，也不适合高效执行排行榜和分数区间查询。跳表天然保持元素有序，能够支持按排名和范围遍历，因此 Redis 将哈希表与跳表组合使用，各自负责擅长的查询方式。

### 2. Zset 中成员分数发生变化时，底层如何处理？

当执行 `ZADD` 更新已有成员的 `score` 时，Redis 需要修改成员对应的分值，并根据新分数调整其在有序结构中的位置。在跳表编码下，通常需要从原排序位置移除后按新分数重新插入，同时保持哈希表中查询到的分值一致。

### 3. Zset 什么时候使用 Listpack，什么时候转换为跳表编码？

当元素数量和成员大小都在配置允许范围内时，Redis 可以使用 `Listpack` 以减少内存占用；一旦新增元素导致集合规模或单个成员长度超过阈值，就会转换为普通的跳表编码。相关阈值可通过 `zset-max-listpack-entries` 和 `zset-max-listpack-value` 控制。

## 常见误区

**误区：** `Zset` 的底层始终只有跳表一种结构。  
**正确理解：** 小型 `Zset` 通常使用紧凑的 `Listpack` 编码；超过阈值后，才会转换为哈希表与跳表组合的普通编码。

**误区：** `Zset` 只靠跳表就完成了所有高效查询。  
**正确理解：** 跳表主要负责排序、排名和范围查询；普通编码下还会结合哈希表，以快速完成按成员查找和分值读取。
