## 答案

## 简要回答

Redis 的虚拟内存 `VM` 是早期将部分 value 换出到磁盘、仅尽量保留 key 在内存中的机制，用于让数据集超过物理内存。该机制已在 Redis `2.4` 弃用，并于 `2.6` 移除，现代 Redis 不再使用它。

## 专业名词解释

### 虚拟内存 VM

这里的 `VM` 是 Redis 早期提供的应用层换页机制，不是当前推荐能力。它允许将暂时不常访问的数据值写入磁盘，在访问时再加载回内存。

### Swap File

Swap File 是 Redis `VM` 用于保存被换出 value 的磁盘文件。被换出的数据需要发生磁盘读取后才能重新参与命令处理。

### 工作集

工作集是某一时间段内真正被频繁访问的热点数据集合。`VM` 的设计前提是：全部 key 可以留在内存，而大量冷 value 很少被访问。

## 面试回答

Redis 的虚拟内存 `VM` 是早期为“数据总量大于内存，但热点数据只占一部分”的场景设计的机制。它与操作系统直接把整个进程页面换出的方式不同：Redis 自己管理换出策略，通常尽量让 key 继续保存在内存中，而把较少访问的 value 写入磁盘 swap 文件。这样查找 key 仍可在内存中快速完成，只有真正访问到被换出的 value 时，才需要从磁盘重新读取。

这种设计依赖一个重要假设：key 集合能够装入内存，并且大多数请求集中访问较小的热点工作集。如果频繁访问的数据已经被换出，查询就会触发磁盘 I/O，Redis 的低延迟优势会明显下降。尤其 Redis 的命令处理强调快速响应，换入换出会带来不可预测的延迟抖动，复杂度和维护成本也较高。

因此，Redis 官方后来放弃了这一方向：`VM` 在 Redis `2.4` 已被弃用，并在 Redis `2.6` 被移除。现代 Redis 仍然定位为以内存访问为核心的数据存储，磁盘主要用于 `RDB`、`AOF` 等持久化，而不是在请求路径上通过 `VM` 承载超出内存的数据。

如果当前业务数据已经超过单机内存，通常应采用更符合现代 Redis 架构的方案：通过过期和淘汰策略只缓存热点数据；优化数据结构和 key 设计降低内存消耗；使用 `Redis Cluster` 分片扩展容量；或者将完整冷数据保存在数据库、对象存储等外部系统中，Redis 只承担热点缓存和实时访问层。

## 核心要点

- `VM` 是 Redis 早期磁盘换页机制
- 它主要将冷 value 换出到磁盘
- key 集合仍需尽量保留在内存中
- 换入磁盘数据会明显增加访问延迟
- `VM` 在 Redis `2.4` 被弃用
- `VM` 在 Redis `2.6` 已被移除

## 面试追问

### 1. Redis 的 VM 与 RDB、AOF 有什么区别？

`VM` 的目的，是在运行期间将部分冷数据值换到磁盘，并可能在业务访问时重新加载，直接影响请求路径。`RDB` 和 `AOF` 的目的则是持久化与故障恢复，它们不会把“查询时去磁盘读取被换出 value”作为正常数据访问模式。

### 2. 为什么 Redis 后来移除了虚拟内存机制？

因为 `VM` 会让访问被换出数据的请求依赖磁盘 I/O，破坏 Redis 以内存访问获得稳定低延迟的核心特性；同时它带来换页、并发和配置管理复杂度。官方最终选择聚焦纯内存访问与持久化，而不是支持超内存数据集的在线换页。

### 3. 现代 Redis 遇到数据量超过内存时应如何处理？

如果 Redis 用作缓存，应只保留热点数据，并配置合理的过期时间与 `maxmemory-policy`；如果需要扩大 Redis 可承载的数据容量，可使用 `Redis Cluster` 进行分片扩容；完整且低频访问的数据通常应交给数据库或其他磁盘型存储保存。

## 常见误区

**误区：** 当前 Redis 仍可以开启 `VM`，通过磁盘扩展内存容量。  
**正确理解：** `VM` 是历史机制，在 Redis `2.4` 已弃用、`2.6` 已移除，现代 Redis 不再提供该能力。

**误区：** Redis 的持久化机制就是虚拟内存，会在访问数据时从磁盘读取 value。  
**正确理解：** `RDB` 和 `AOF` 主要用于数据恢复与持久化；正常服务期间，现代 Redis 的在线数据访问仍以内存为核心。
