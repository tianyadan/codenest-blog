## 答案

## 简要回答

EMBSTR 的 `44` 字节阈值来自内存布局优化：在 64 位环境下，redisObject、sdshdr8、字符串内容和结尾 `\0` 恰好可放入 `jemalloc` 的 `64` 字节分配区间。Redis 早期阈值曾是 `39`，在 SDS 头部压缩后提升为 44。

## 专业名词解释

### EMBSTR

`EMBSTR` 是 Redis 保存短字符串的一种内部编码方式。它将对象头和字符串内容放在同一块连续内存中，只需一次内存分配和释放，访问局部性也更好。

### SDS

`SDS` 是 Redis 自定义的动态字符串结构，除实际字符内容外，还维护长度、可用空间等元数据。不同版本的 SDS 头部大小变化，会影响 `EMBSTR` 能容纳的字符串长度。

### jemalloc

`jemalloc` 是 Redis 常用的内存分配器，它按若干固定大小区间分配内存。`EMBSTR` 的阈值设计目标之一，就是让整个短字符串对象落在较小的分配区间内。

## 面试回答

Redis 的 `String` 对象除了整数可使用 `int` 编码外，普通短字符串通常可使用 `embstr`，较长字符串则使用 `raw`。`embstr` 的核心优化是把 `redisObject` 和承载内容的 `SDS` 放在一次申请的连续内存中；而 `raw` 通常需要分别为对象头和 `SDS` 分配内存。

在常见 64 位构建中，Redis 源码中的 `OBJ_ENCODING_EMBSTR_SIZE_LIMIT` 为 `44`。原因是 `embstr` 对象希望完整落入 `jemalloc` 的 `64` 字节分配区间：`redisObject` 占 `16` 字节，短字符串使用的 `sdshdr8` 占 `3` 字节，C 字符串末尾还需要 `1` 字节的 `\0`，因此字符串内容最多为：

```text
64 - 16 - 3 - 1 = 44
```

这样，长度不超过 `44` 字节的字符串可以保持一次分配、一次释放，并减少额外指针跳转和缓存未命中；超过该长度后，即使继续使用连续分配也会进入更大的内存分配区间，收益降低，因此转为 `raw` 更合适。

这个阈值并非一直是 `44`。在 Redis `3.0` 中，源码使用的阈值是 `39`，因为当时 `SDS` 头部采用包含 `len` 和 `free` 的旧结构，头部占 `8` 字节：

```text
64 - 16 - 8 - 1 = 39
```

到了 Redis `3.2`，SDS 改为按容量选择更紧凑的头部结构，短字符串的 `sdshdr8` 仅需 `3` 字节，于是 `EMBSTR` 阈值从 `39` 提升到 `44`。因此，阈值调整的本质不是业务配置变化，而是 **SDS 元数据布局优化后，在同一个 64 字节内存区间中能够容纳更多有效字符串内容**。

**还要注意两点**：第一，`44` 指的是字符串内容的字节数，而不是字符个数，UTF-8 中文通常会占多个字节；第二，`embstr` 适合不可变短字符串，一旦执行会修改内容的操作，Redis 可能将其转换为 `raw` 编码。

## 核心要点

- `EMBSTR` 将对象头和字符串连续存储
- `44` 来自 `64` 字节分配区间计算
- 公式为 `64 - 16 - 3 - 1 = 44`
- Redis `3.0` 中阈值曾为 `39`
- Redis `3.2` 因 SDS 头部压缩提升为 `44`
- 阈值按字节计算，不按字符数量计算

## 面试追问

### 1. EMBSTR 和 RAW 编码的主要区别是什么？

`EMBSTR` 会将 `redisObject` 与 `SDS` 放在同一块连续内存中，通常只进行一次内存分配和释放，适合短且基本不修改的字符串。`RAW` 中对象头和字符串数据通常分别分配，更适合较长或后续可能扩展、修改的字符串。

### 2. 为什么 Redis 不把 EMBSTR 阈值继续调大，让更多字符串享受一次分配？

因为阈值设计不是越大越好，而是要匹配内存分配器的分配区间。超过 `44` 字节后，完整对象无法继续落入 `64` 字节区间，需要申请更大的内存块，可能造成额外空间浪费；同时较长字符串后续修改和扩容的可能性也更高，使用 `raw` 更合理。

### 3. 为什么一个看起来不长的中文字符串可能无法使用 EMBSTR？

因为 Redis 判断阈值使用的是字节长度，而不是字符数量。在 UTF-8 编码下，一个常见中文字符通常占 `3` 个字节，因此十几个中文字符加上其他内容就可能超过 `44` 字节，最终使用 `raw` 编码。可以通过 `STRLEN` 查看实际字节长度。

## 常见误区


**误区：** 字符串长度不超过 `44` 个字符，就一定会使用 `EMBSTR`。  
**正确理解：** 阈值按字节计算，且可被编码优化或后续修改操作影响；例如可转换为整数的值可能使用 `int` 编码，中文字符串也可能因字节数超过限制而使用 `raw`。
