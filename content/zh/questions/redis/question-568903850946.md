## 答案


Redis Big Key 指的是 value 特别大的 key，或者集合元素数量特别多的 key。

比如：
一个 String 存了几 MB 数据。
一个 Hash 里有几十万个 field。
一个 List / Set / ZSet 里有几十万个元素。

**Big Key 的问题主要有：**
1 ） 占用内存大
2 ） 网络传输慢
3 ） 删除时阻塞 Redis
4 ） 影响主从同步
5 ） 容易导致 Redis 延迟抖动
6 ） 集群模式下可能造成数据倾斜

解决思路是：
1 ） 拆分 Big Key
2 ） 控制单个 key 的大小
3 ） 大集合分页读取
4 ） 删除时使用 UNLINK 异步删除
5 ） 对 Hash、Set、ZSet、List 这类集合 key 分批删除
6 ） 建立监控，定期扫描 Big Key

**发现 Big Key 常用方式：**
1 ） redis-cli –bigkeys
2 ） MEMORY USAGE key
3 ） SCAN 渐进式扫描
4 ） 业务侧埋点统计 value 大小
5 ） Redis 监控平台告警

**删除 Big Key 的正确姿势：**
不要直接 DEL。
优先用 UNLINK。
如果是大集合类型，要先分批删除里面的元素，最后再删除 key。

扩展知识：

**一、什么是 Big Key？**

Big Key 不是指 key 的名字很长。

而是指：key 对应的 value 太大。

比如：
String 类型 ： 一个 key 存了 5MB、10MB 的 JSON 字符串。
Hash 类型：一个 Hash 里面存了几十万个 field。
List 类型：一个 List 里堆了几十万条消息。
Set 类型：一个 Set 里存了几十万个用户 ID。
ZSet 类型：一个排行榜 ZSet 里存了几百万个元素。
这些都属于 Big Key。

————————————————————

**二、Big Key 会带来什么问题？**

1 ）内存占用过大：一个 Big Key 会占用大量内存。如果很多 Big Key 堆在一起，很容易造成 Redis 内存压力。
2 ） 网络传输慢 ：比如你执行：GET big:string ，如果这个 value 有 10MB ，Redis 要把 10MB 数据一次性返回给客户端。这会导致网络 IO 变慢，客户端也会卡。
3 ） Redis 主线程阻塞 ：Redis 的命令执行主要还是单线程模型。如果你对 Big Key 执行：DEL 或者：HGETALL 或者：SMEMBERS 这类命令，就可能让 Redis 卡住 ，因为它要一次性处理大量数据。
4 ） 主从同步压力大 ：Big Key 被修改后，主从复制需要同步大量数据 ，如果网络抖动或者数据太大，可能导致复制延迟。
5 ） 集群数据倾斜 ：Redis Cluster 下，一个 key 只能落到一个 slot，如果某个 Big Key 特别大，它所在的节点内存压力就会明显高于其他节点 ，这就是数据倾斜。


————————————————————

**三、常见 Big Key 场景**

1 ） 把大量对象塞进一个 JSON 字符串
例如：user:all:list -> 巨大的用户列表 JSON 这种设计很危险。
2 ） 热门文章点赞用户全部放到一个 Set
例如： article:like:userIds:1001 如果文章爆了，Set 可能变成几十万元素。
3 ） 排行榜无限增长
例如： rank:article:hot 如果不做时间窗口和数量控制，ZSet 会越来越大。
4 ） 消息队列积压
例如：List 里堆了大量待消费消息。


————————————————————

**四、如何解决 Big Key？**

核心原则：不要让一个 key 承担太多数据。

1 ）拆分 key
比如原来：user:all里面放全部用户。可以拆成：user:page:1、user:page:2、user:page:3
或者按业务 ID 分片：user:list:0、user:list:1、user:list:2
2 ） 大 Hash 拆小 Hash
原来：article:view:hash 里面存所有文章访问量。可以改成：article:view:hash:0、article:view:hash:1...
3 ） 大集合控制数量
比如排行榜：只保留 Top 1000。超过就定期裁剪。
ZSet 可以用：ZREMRANGEBYRANK ，清理低排名数据。
4 ） 大列表分页读取
不要：LRANGE key 0 -1 而是：LRANGE key 0 99 、LRANGE key 100 199 也就是分页读取。
5 ） 避免 HGETALL、SMEMBERS 这种全量命令
大 key 场景下，尽量不用：HGETALL、SMEMBERS ...  应该改用：HSCAN、SSCAN ... 分批扫描。

————————————————————

**五、如何发现 Big Key？**

1 ） redis-cli –bigkeys
这是 Redis 自带工具。

示例：redis-cli -h 127.0.0.1 -p 6379 –bigkeys 它会扫描 Redis 中比较大的 key。
优点： 简单方便。
缺点：只能给你一个大概结果，不一定足够精细。

2 ） MEMORY USAGE
可以查看某个 key 占用的内存。
例如：MEMORY USAGE user:info:1001 ，它会返回这个 key 大概占用的字节数。适合定位单个 key 的内存占用。

3 ） SCAN + TYPE + MEMORY USAGE
生产环境更推荐这种方式，用 SCAN 渐进式扫描 key。不要用 KEYS * ，因为 KEYS 会阻塞 Redis。
流程大概是： SCAN 扫描 key 、TYPE 判断类型、MEMORY USAGE 计算内存 超过阈值就记录下来

4 ） 业务侧统计
比如写入 Redis 前，记录：value 大小、集合元素数量、key 类型、过期时间，这样可以从源头发现问题。这个方案很工程化。


**六、删除 Big Key 的正确姿势**
Big Key 千万不要随便 DEL ，因为 DEL 是同步删除。如果 key 很大，Redis 主线程会被阻塞。

错误姿势：DEL big:key 如果这个 key 里面有几十万个元素，Redis 可能会明显卡顿。


正确姿势一：
使用 UNLINK ： UNLINK big:key ， UNLINK 会把 key 从 Redis keyspace 里移除，然后后台异步释放内存。相比 DEL，它对主线程影响更小。适合删除大 String、大对象、大集合。

正确姿势二：
集合类型分批删除，如果是 Hash：使用 HSCAN 分批扫描 field。然后每次 HDEL 一小批。比如每次删 500 个 field。最后再 UNLINK 整个 key。
如果是 Set：使用 SSCAN 分批扫描 member。然后每次 SREM 一小批。最后再 UNLINK。
如果是 ZSet：使用 ZSCAN 分批扫描 member。然后每次 ZREM 一小批。最后再 UNLINK。
如果是 List：可以用 LTRIM 逐步裁剪。或者用业务方式分批消费。最后再 UNLINK。
