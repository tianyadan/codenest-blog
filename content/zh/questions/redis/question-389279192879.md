## 答案

Redis **支持事务**，但它和 MySQL 的事务机制并不相同。

Redis 事务的核心目标是：

* 保证命令按顺序执行
* 保证执行期间不会被其他客户端命令插入

但 Redis 并不支持传统数据库那种完整的事务回滚能力。

Redis 事务主要依赖以下四个命令实现：

* MULTI
* EXEC
* WATCH
* DISCARD

---

**Redis 事务执行流程**

1 ） 执行 MULTI 开启事务

开启后，后续发送的命令不会立即执行，而是暂时进入事务队列等待。

——————————————————

2 ） 命令进入队列

事务期间可以连续提交多条命令。

Redis 会返回：

QUEUED

表示当前命令已经成功加入事务队列。

——————————————————

3 ） 执行 EXEC 提交事务

执行 EXEC 后，Redis 会一次性按照顺序执行队列中的所有命令。

整个执行过程具备原子性，不会被其他客户端命令打断。

——————————————————

4 ） 执行 DISCARD 取消事务

如果事务还未提交，可以通过：

DISCARD

直接清空当前事务队列，放弃执行。

——————————————————

5 ） 使用 WATCH 监控 key

WATCH 可以在事务开始前监听某些 key。

如果这些 key 在事务提交前被其他客户端修改：

EXEC

将不会执行事务。

这种机制本质上属于：乐观锁（Optimistic Lock）思想，主要用于解决并发修改问题。

---

**Redis 事务特点**

优点：

* 保证命令顺序执行
* 执行期间不会被其他客户端插队
* 实现简单，性能较高

缺点：

* 不支持真正意义上的事务回滚
* 某条命令失败不会撤销已执行命令
* 不满足完整 ACID 特性

---

**一句话总结：**

Redis 事务更像是：

“批量顺序执行命令”,而不是传统关系型数据库中的完整 ACID 事务。
