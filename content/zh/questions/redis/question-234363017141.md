## 答案

## 简要回答

Redis 主从复制常见拓扑包括**一主一从**、**一主多从**、**树状**或**链式复制**，以及在分片场景下每个主节点配置从节点的集群拓扑。不同拓扑主要在读扩展、故障恢复和主节点复制压力之间取舍。

## 专业名词解释

### 主从复制

主节点负责处理写入并将数据变更异步同步给从节点；从节点通常用于数据冗余、故障切换或承担读请求。

### 级联复制

级联复制是指从节点继续作为其他从节点的复制源，形成树状或链式结构，用于降低主节点直接连接大量从节点的同步压力。

### 分片

分片是将不同 key 分散存放到多个主节点中，每个主节点只负责一部分数据。它解决的是容量和写吞吐扩展问题，不等同于主从复制。

## 面试回答

Redis 主从复制最基础的拓扑是**一主一从**：一个主节点负责写入，一个从节点复制主节点数据。这种结构部署简单，适合数据备份和基础故障恢复，但读扩展与高可用能力有限。

第二种是**一主多从**：一个主节点挂载多个从节点。写请求仍进入主节点，多个从节点可用于读扩展、备份或配合 `Sentinel` 进行自动故障转移。这是比较常见的高可用部署方式，但从节点越多，主节点在全量同步、网络发送和复制维护上的压力越大。

第三种是**树状或链式复制**：主节点只直接同步给部分从节点，这些从节点再同步给下游从节点。例如 `Master -> Replica A -> Replica B`。这种结构能够分散主节点的复制压力，适合跨机房分发或从节点数量较多的场景；但复制链路变长后，下游数据延迟会增加，上游从节点故障也可能影响其下游复制。

在数据规模和写吞吐需要水平扩展时，通常使用**分片加副本**的拓扑，例如 `Redis Cluster` 中存在多个主节点，每个主节点负责一部分 slot，并各自配置一个或多个从节点。这样，主节点之间负责扩展容量和写能力，从节点负责副本冗余与故障切换。

需要强调的是，Redis 复制默认是异步的，从节点可能存在复制延迟，故障切换时也存在少量已确认写入丢失的风险。因此，拓扑选择不能只看节点数量，还要结合读写比例、容灾要求、跨机房延迟和可接受的数据一致性风险。

## 核心要点

- 一主一从适合基础备份与容灾
- 一主多从可扩展读取与提升可用性
- 级联复制可降低主节点同步压力
- 链路越长，下游复制延迟越明显
- `Redis Cluster` 通常采用分片加副本
- Redis 复制默认是异步复制

## 面试追问

### 1. 一主多从与树状复制如何选择？

一主多从结构简单，故障切换和运维更直观，适合从节点数量不多、主节点网络压力可控的场景。树状复制能够减少主节点直接向大量从节点同步数据的压力，但会增加拓扑复杂度和下游复制延迟，通常在跨机房或副本规模较大时考虑。

### 2. 主从复制与 Redis Cluster 有什么区别？

主从复制是把同一份数据复制到副本节点，主要解决数据冗余、读扩展和故障恢复问题；`Redis Cluster` 会将 key 按 slot 分散到多个主节点，主要解决数据容量和写吞吐扩展问题。集群中的每个主分片通常还会配置从节点，两者可以组合使用。

### 3. 有了从节点后，主节点故障是否一定不会丢数据？

不一定。Redis 主从复制默认采用异步方式，主节点可能已经向客户端返回写入成功，但该变更还没有同步到从节点。如果此时主节点故障并提升一个尚未收到该写入的从节点，最近的一部分数据仍可能丢失。

## 常见误区

**误区：** 从节点越多越好，增加副本不会影响主节点性能。  
**正确理解：** 更多从节点能够增强读扩展和数据冗余能力，但也会增加主节点的复制连接、网络发送以及全量同步压力。

**误区：** `Redis Cluster` 就是把一个主节点复制成多个从节点。  
**正确理解：** `Redis Cluster` 的核心是分片，将不同数据分配给多个主节点；副本复制是在每个分片内部用于高可用的配套机制。
