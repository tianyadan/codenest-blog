## 答案

## 简要回答

Redis 的 `String` 并不是统一直接存为 C 字符串，而是由 `redisObject` 包装，并根据值的内容和长度选择 `int`、`embstr` 或 `raw` 编码；其中字符串内容通常基于 `SDS` 保存。

## 专业名词解释

### SDS

`SDS` 是 Redis 自定义的动态字符串结构，会记录已使用长度和可用空间。相比以 `\0` 判断结束的 C 字符串，它能以常数时间获取长度，并且是二进制安全的，可以保存图片、序列化数据等字节内容。

### 编码方式

编码方式表示同一种 Redis 数据类型在内存中的具体存储形式。`String` 类型会根据值是否为整数、字符串长度等条件，选择更节省内存或更适合修改的实现。

## 面试回答

Redis 的 `String` 类型底层由 `redisObject` 统一封装，具体存储形式不是固定的，而是有三种常见编码：`int`、`embstr` 和 `raw`。

如果字符串值可以安全转换为 64 位有符号整数，例如 `"100"`，Redis 可以使用 `int` 编码，直接保存整数值，减少字符串对象带来的内存开销。

如果值是较短的字符串，Redis 通常采用 `embstr` 编码，把 `redisObject` 和保存内容的 `SDS` 分配在一块连续内存中。这样只需要一次内存分配，访问局部性也更好，适合创建后修改较少的短字符串。常见 Redis 版本中，其长度阈值通常为 44 字节，但面试中应说明具体阈值以实际版本实现为准。

如果字符串较长，或者短字符串发生修改后不再适合嵌入式存储，就会使用 `raw` 编码。此时 `redisObject` 与 `SDS` 分开分配，后续扩容和修改更灵活。

因此，更准确的表述是：Redis `String` 的字符串内容主要基于 `SDS`，同时通过不同编码策略优化整数值、短字符串和长字符串的内存占用与操作效率。可以使用 `OBJECT ENCODING key` 查看某个键当前的内部编码。

## 核心要点

- `String` 由 `redisObject` 统一封装
- 字符串内容通常由 `SDS` 保存
- 整数值可能使用 `int` 编码
- 短字符串通常使用 `embstr` 编码
- 长字符串或修改后对象通常使用 `raw`
- 可用 `OBJECT ENCODING` 查看编码

## 面试追问

### 1. Redis 为什么不用普通 C 字符串，而要使用 SDS？

普通 C 字符串依赖 `\0` 判断结尾，获取长度需要遍历，也不适合直接保存包含空字节的二进制内容。`SDS` 会记录长度和容量，既能常数时间获取长度，又支持二进制安全存储，并能通过预分配等方式降低频繁修改时的内存开销。

### 2. embstr 和 raw 有什么区别？

`embstr` 会将 `redisObject` 与 `SDS` 放在连续内存中，分配次数少，适合较短且修改不频繁的字符串；`raw` 中两者分开分配，更适合较长字符串或需要修改、扩容的场景。短字符串发生修改后，也可能转换为 `raw`。

### 3. 如何查看一个 String 键实际使用了哪种编码？

可以执行 `OBJECT ENCODING key` 查看内部编码，返回值可能是 `int`、`embstr` 或 `raw`。需要注意，编码由 Redis 根据值内容、长度以及后续操作自动选择和转换，业务代码不应依赖某个键永久保持某种编码。

## 常见误区

**误区：** Redis 的 `String` 底层实现就是 `SDS`，没有其他形式。  
**正确理解：** `String` 由 `redisObject` 封装，字符串内容常基于 `SDS`，但整数值还可能直接使用 `int` 编码。

**误区：** 只要字符串长度较短，就会永远保持 `embstr` 编码。  
**正确理解：** `embstr` 适合短字符串的初始存储，但发生修改或不再满足内部条件后，编码可能转换为 `raw`。
