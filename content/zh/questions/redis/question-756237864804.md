## 答案

## 简要回答

`Sentinel` 主要解决单主复制架构的监控与自动故障转移，不负责数据分片；`Redis Cluster` 同时提供数据分片与分片级高可用，适合容量和写吞吐需要横向扩展的场景。

## 专业名词解释

### Sentinel

`Sentinel` 是 Redis 的高可用组件，负责监控主从节点、判断主节点故障、自动将从节点提升为新主节点，并帮助客户端发现当前主节点地址。

### Redis Cluster

`Redis Cluster` 是 Redis 的分布式集群方案，将 key 分配到多个主节点管理的 `16384` 个 hash slot 中，并可为每个主节点配置从节点用于故障切换。

### Hash Slot

Hash Slot 是 `Redis Cluster` 进行数据分片的逻辑槽位。一个 key 会被映射到某个 slot，再由负责该 slot 的主节点保存和处理。

## 面试回答

`Sentinel` 模式和 `Redis Cluster` 模式都能提升 Redis 的可用性，但解决的问题不同。

`Sentinel` 通常基于一主多从架构运行。数据仍然完整存放在一个主节点及其从节点上，`Sentinel` 负责监控、通知、自动故障转移和主节点发现。当主节点不可用时，多个 `Sentinel` 达成判断后，会选择一个从节点升级为新的主节点，并通知客户端切换连接。它解决的是**单份数据的高可用问题**，不会把数据拆分到多个主节点，因此整体容量和写吞吐仍主要受单主节点限制。

`Redis Cluster` 则解决**分片扩容与高可用**两个问题。它将全部 key 按 slot 分散到多个主节点，每个主节点负责一部分数据，并通常配置从节点作为副本。某个主节点故障时，其从节点可以被提升；当容量或吞吐不足时，还可以增加主节点并迁移 slot，从而横向扩展存储能力和写入能力。

两者对客户端和命令使用的影响也不同。`Sentinel` 模式下，客户端主要需要感知主节点切换，数据没有跨分片限制；`Redis Cluster` 客户端需要识别 slot 路由和重定向。对于 `MGET`、事务或 Lua 脚本等涉及多个 key 的操作，Cluster 中相关 key 必须位于同一个 slot，通常通过 hash tag，例如 `order:{1001}:detail` 与 `order:{1001}:status`，保证共同操作的 key 落在一起。

选择时，如果数据规模和写吞吐单节点能够承载，主要诉求是自动容灾和部署简单，优先考虑 `Sentinel`；如果数据容量、写吞吐或业务增长已经需要水平扩展，则选择 `Redis Cluster`。需要注意，两者都基于异步复制实现故障切换，极端故障情况下都不能天然保证零数据丢失。

## 核心要点

- `Sentinel` 重点解决自动故障转移
- `Sentinel` 不提供数据分片能力
- `Redis Cluster` 提供分片与高可用
- Cluster 使用 `16384` 个 hash slot
- Cluster 可横向扩展容量和写吞吐
- 两种模式都不天然保证零数据丢失

## 面试追问

### 1. Sentinel 模式为什么不能解决写吞吐和容量扩展问题？

`Sentinel` 管理的是同一份数据的主从副本，所有写入仍然由当前主节点处理，从节点主要承担副本和可选读流量。因此增加从节点不能分摊主节点的写入和数据容量压力，单主瓶颈仍然存在。

### 2. Redis Cluster 中为什么多 key 操作会受到限制？

因为不同 key 可能映射到不同 slot，并由不同主节点保存，单个节点无法原子处理分散在多个节点上的命令。Cluster 只支持相关 key 位于同一 slot 的多 key 操作，可以通过 `{...}` hash tag 将需要共同操作的 key 固定到同一 slot。

### 3. 单机数据量不大，但要求主节点故障后自动恢复，应选择哪种模式？

通常选择主从复制配合 `Sentinel` 更合适。它能提供主节点监控、自动故障转移和服务发现，同时部署与客户端适配成本通常低于 Cluster。只有当容量或写吞吐需要按多个主节点扩展时，再考虑 `Redis Cluster`。

## 常见误区

**误区：** `Sentinel` 只是监控工具，不能自动完成主从切换。  
**正确理解：** `Sentinel` 不仅负责监控和通知，也能够在主节点故障时发起自动故障转移，将从节点提升为新的主节点。

**误区：** 使用 `Redis Cluster` 后，所有多 key 命令、事务和 Lua 脚本都能像单机模式一样任意执行。  
**正确理解：** Cluster 中涉及多个 key 的操作通常要求这些 key 位于同一个 slot，应通过合理的 key 设计和 hash tag 处理相关业务数据。
