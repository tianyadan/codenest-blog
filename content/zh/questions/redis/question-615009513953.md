## 答案

## 简要回答

Redis 的 Geo 是用于存储经纬度并执行位置查询的地理空间索引能力。它底层基于 `Sorted Set` 和 `GeoHash` 编码实现，适合附近门店、骑手定位、距离计算等场景。

## 专业名词解释

### GeoHash

`GeoHash` 是一种将二维经纬度坐标编码为可比较数值或字符串的方式。位置越接近，编码通常越相似，便于按区域筛选附近元素。

### 地理空间索引

地理空间索引是针对坐标数据建立的查询能力，用于快速完成附近位置搜索、范围过滤和距离计算，而不是遍历全部位置逐个比较。

## 面试回答

Redis Geo 可以理解为 Redis 提供的一组地理位置操作命令，用来保存带有经纬度的成员，并支持位置查询。它并不是独立的底层存储类型，实际数据存储在 `Sorted Set` 中：`GEOADD` 会将成员的经纬度通过 `GeoHash` 编码成分值，再写入有序集合。

常见操作包括：使用 `GEOADD` 添加或更新位置，使用 `GEOPOS` 获取成员坐标，使用 `GEODIST` 计算两个成员之间的距离，使用 `GEOSEARCH` 按圆形半径或矩形范围查询附近成员，并可结合 `WITHDIST`、`ASC`、`COUNT` 返回最近的若干结果。

```bash
GEOADD shop:locations 120.3826 36.0671 shop:1001
GEOADD shop:locations 120.4012 36.0754 shop:1002

GEODIST shop:locations shop:1001 shop:1002 km

GEOSEARCH shop:locations \
  FROMLONLAT 120.3900 36.0700 \
  BYRADIUS 3 km \
  WITHDIST ASC COUNT 10
```

由于底层是 `Sorted Set`，删除某个地理成员可以直接使用 `ZREM`。在新代码中，范围查询通常优先使用 `GEOSEARCH`，而不是较早的 `GEORADIUS` 系列命令。

Geo 适合门店检索、附近车辆、配送范围初筛等普通位置业务，但它主要解决简单的位置范围和距离查询，不适合复杂 GIS 分析、精密测绘或多维空间运算场景。另外，写入坐标时参数顺序是**经度在前、纬度在后**，这是实践中很容易写反的地方。

## 核心要点

- Geo 用于存储坐标并执行附近查询
- 底层通过 `Sorted Set` 保存数据
- 经纬度会编码为 `GeoHash` 分值
- `GEOSEARCH` 可按半径或矩形搜索
- 删除成员可直接使用 `ZREM`
- 坐标参数顺序是经度在前

## 面试追问

### 1. Redis Geo 为什么可以使用 ZREM 删除位置数据？

因为 Geo 的成员实际上保存在 `Sorted Set` 中，成员名仍然是有序集合中的 member，只是 score 存储的是由经纬度转换得到的地理编码。因此删除地理成员本质上就是删除有序集合成员，可以直接执行 `ZREM key member`。

### 2. GEOSEARCH 与 GEODIST 的作用有什么区别？

`GEODIST` 用于计算同一个 Geo key 中两个已存在成员之间的距离；`GEOSEARCH` 用于以某个坐标或已有成员为中心，查询指定半径或矩形范围内的多个成员，并可返回距离、坐标及排序结果。

### 3. Redis Geo 适合所有地图位置业务吗？

不适合。Redis Geo 很适合附近门店、车辆位置和配送范围初筛等简单场景，但距离计算基于近似地球模型，且查询能力有限。涉及复杂地理关系、道路路径规划或高精度空间分析时，通常需要专业 GIS 或地图服务配合完成。

## 常见误区

**误区：** Redis Geo 是一种与 `Sorted Set` 完全独立的底层数据结构。  
**正确理解：** Redis Geo 是建立在 `Sorted Set` 之上的地理空间操作能力，经纬度经过编码后作为有序集合分值存储。

**误区：** `GEOADD` 的坐标参数可以按照常见的“纬度、经度”顺序传入。  
**正确理解：** `GEOADD` 的参数顺序是先经度、后纬度，写反后会导致位置错误，超出支持范围时还会直接报错。
