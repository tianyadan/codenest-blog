## 答案

## 简要回答

在 `Redis Cluster` 中，客户端先根据 key 计算 hash slot：`CRC16(key) % 16384`，再根据 slot 与节点的映射关系将请求发送到对应主节点。若 key 含有效 hash tag，则只对 `{}` 中的内容计算 slot。

## 专业名词解释

### Hash Slot

Hash Slot 是 Redis Cluster 的逻辑分片单位，固定共有 `16384` 个槽位。每个主节点负责其中一部分槽位，key 最终通过槽位定位到节点。

### Hash Tag

Hash Tag 是 key 中 `{}` 包裹的有效子串。例如 `order:{1001}:detail` 与 `order:{1001}:status` 都只对 `1001` 计算 slot，因此会落到同一个槽位。

### 重定向

客户端将请求发送到错误节点，或槽位正在迁移时，Redis 会返回重定向响应。常见类型是 `MOVED` 和 `ASK`，客户端需要据此转发请求或更新路由信息。

## 面试回答

在 `Redis Cluster` 中，数据不是按 key 直接映射到节点，而是先映射到 hash slot，再由 slot 找到负责处理该槽位的主节点。集群固定有 `16384` 个 slot，普通 key 的计算方式是：

```text
slot = CRC16(key) % 16384
```

例如可以通过命令查看某个 key 所属的槽位：

```bash
CLUSTER KEYSLOT order:1001
```

如果 key 中存在有效的 hash tag，例如 `order:{1001}:detail`，Redis 只会对 `{1001}` 中的内容计算 slot。这样可以让多个相关 key 落到同一个槽位，从而支持同 slot 下的 `MGET`、事务或 Lua 脚本等多 key 操作。

```bash
CLUSTER KEYSLOT order:{1001}:detail
CLUSTER KEYSLOT order:{1001}:status
```

客户端通常会维护一份 slot 到节点的路由表，路由信息可以从集群节点获取，例如使用 `CLUSTER SHARDS`；一些客户端或兼容场景也会使用 `CLUSTER SLOTS` 获取槽位分布。发送命令时，客户端先计算 key 的 slot，再把请求直接发到负责该 slot 的节点，避免每次请求都向服务端查询位置。

如果客户端路由表过期，请求被发送到错误节点，节点会返回 `MOVED` 重定向，表示该 slot 已稳定归属于另一个节点，客户端应更新路由并重新发送请求。若 slot 正处于迁移过程，可能返回 `ASK`，客户端只需临时将当前请求转发到目标节点，并在请求前发送 `ASKING`，不能简单把它当作永久路由变化。

因此，定位过程可以概括为：**计算 key 的 slot -> 查询或缓存 slot 所属节点 -> 发送请求 -> 根据 `MOVED` 或 `ASK` 处理路由变化**。

## 核心要点

- Cluster 固定使用 `16384` 个 slot
- 普通 key 按 `CRC16(key) % 16384` 定位
- Hash tag 可让相关 key 落入同一 slot
- 客户端通常缓存 slot 与节点映射
- `MOVED` 表示应更新长期路由
- `ASK` 表示槽位迁移中的临时转发

## 面试追问

### 1. 为什么 Redis Cluster 使用 hash tag？

因为多个 key 默认可能被分配到不同 slot，导致多 key 命令、事务或 Lua 脚本无法在单个节点完成。通过设计形如 `cart:{1001}:items`、`cart:{1001}:total` 的 key，可以保证相关数据位于同一 slot，支持共同操作。

### 2. `MOVED` 和 `ASK` 重定向有什么区别？

`MOVED` 表示某个 slot 已经由新的节点稳定负责，客户端应更新本地路由表并将后续请求发送到新节点。`ASK` 通常发生在 slot 迁移过程中，只要求当前请求临时访问目标节点，并在发送命令前执行 `ASKING`，不代表槽位归属已经永久变更。

### 3. 客户端为什么不每次都先向集群查询 key 所在节点？

如果每次操作前都额外查询路由，会增加网络往返并降低性能。成熟的 Cluster 客户端通常在初始化时获取 slot 分布并缓存路由，正常请求直接发送到目标节点；只有遇到 `MOVED`、拓扑变化或连接异常时，才更新路由信息。

## 常见误区

**误区：** Redis Cluster 使用一致性哈希直接把 key 映射到某台节点。  
**正确理解：** Redis Cluster 使用固定的 `16384` 个 hash slot，key 先映射到 slot，再由 slot 的归属关系定位具体节点。

**误区：** 收到 `ASK` 响应后，客户端应永久将该 slot 的路由更新到目标节点。  
**正确理解：** `ASK` 通常表示槽位仍在迁移过程中，只要求临时转发当前请求；稳定路由变化应以 `MOVED` 或重新拉取拓扑信息为依据。
