## 答案

## 简要回答

Redis `Zset` 使用跳表而不是红黑树，主要因为两者平均性能同级，但跳表实现更简单，范围遍历和排名扩展更自然；而 `B+` 树更适合磁盘页和块设备访问，不是纯内存有序集合的首选。

## 专业名词解释

### 跳表

跳表是在有序链表上增加多级随机索引的数据结构。它能够快速跳过大量节点，查询、插入和删除的平均时间复杂度为 `O(log N)`，同时保留链表顺序遍历能力。

### 红黑树

红黑树是一种自平衡二叉搜索树，通过节点着色和旋转维持近似平衡，使查询、插入和删除的时间复杂度稳定为 `O(log N)`。

### B+ 树

`B+` 树是多路平衡树，一个节点可以保存多个索引项，能够减少磁盘或页存储场景下的访问次数，因此广泛用于数据库索引和文件系统。

## 面试回答

Redis `Zset` 在数据量较大时使用**哈希表 + 跳表**：哈希表负责根据 `member` 快速查找 `score`，跳表负责按 `score` 排序、范围查询和排名操作。这里选择跳表，而不是红黑树或 `B+` 树，核心原因不是红黑树做不到，而是跳表更贴合 Redis 的使用场景。

首先，跳表和红黑树在查询、插入、删除上的复杂度都能达到 `O(log N)` 级别，性能目标基本一致。但跳表依赖随机层级维护索引，插入和删除只需修改局部指针，不需要红黑树那样处理旋转、变色和复杂的平衡规则，实现与调试成本更低。

其次，`Zset` 经常执行 `ZRANGE`、`ZREVRANGE`、`ZRANGEBYSCORE` 等连续范围遍历。跳表底层本身就是有序链表，定位起点后沿前向或后向指针顺序扫描即可，处理正序、倒序和区间结果都比较直接。Redis 还在跳表节点层级中维护 `span`，用于支持 `ZRANK`、`ZREVRANK` 这类排名查询。

红黑树同样能实现排序和范围查询，但若要高效支持排名，需要额外维护子树元素数量等增强字段，并在旋转、插入、删除时同步更新，工程实现更复杂。

至于 `B+` 树，它的优势主要体现在磁盘或页式存储场景：通过较高的分支因子减少磁盘 I/O 次数，并方便按叶子页范围扫描。Redis 的 `Zset` 数据主要在内存中操作，不需要以降低磁盘页访问次数为首要目标，因此采用 `B+` 树的收益不明显，反而会增加节点管理和实现复杂度。

因此，Redis 选择跳表，本质上是在**足够好的时间复杂度、优秀的范围遍历能力、易于支持排名以及实现简单性**之间取得平衡，而不是因为红黑树或 `B+` 树无法实现有序集合。

## 核心要点

- 跳表与红黑树都可达到 `O(log N)` 级性能
- 跳表插入删除不需要旋转和平衡修复
- 跳表天然适合范围查询和顺序遍历
- `span` 可支持高效排名查询
- 红黑树支持排名需要额外增强维护
- `B+` 树更适合磁盘或页式索引场景

## 面试追问

### 1. Zset 为什么还需要哈希表，只有跳表不行吗？

跳表适合维护排序、范围读取和排名查询，但如果只根据 `member` 查询分值，仍需要沿排序结构查找。Redis 同时维护哈希表，可以快速根据成员定位 `score`；跳表则负责顺序和范围能力，两者组合能够覆盖 `Zset` 的主要访问方式。

### 2. 红黑树也支持有序遍历，为什么说跳表更适合 Redis？

红黑树确实能够实现有序集合，时间复杂度也很好。但 Redis `Zset` 需要频繁做区间遍历、倒序遍历和排名查询。跳表基于链表顺序访问更直接，并可通过层级跨度支持排名；实现增删逻辑也比维护增强红黑树更简单。

### 3. 为什么数据库索引常用 B+ 树，而 Redis Zset 不优先使用它？

数据库索引通常需要访问磁盘页，`B+` 树单个节点容纳多个索引项，可以减少树高和磁盘 I/O，优势明显。Redis `Zset` 的主要操作发生在内存中，不存在同样强烈的磁盘页访问优化需求，因此更倾向选择结构轻量、范围访问方便的跳表。

## 常见误区

**误区：** Redis 选择跳表是因为红黑树无法实现范围查询或排名。  
**正确理解：** 红黑树可以实现这些能力，但通常需要额外维护增强信息；跳表在 Redis 场景下实现更简单、遍历更自然。

**误区：** `B+` 树性能一定优于跳表，因此 Redis 的选择只是历史原因。  
**正确理解：** 数据结构选择取决于访问介质和业务操作。`B+` 树主要优势在磁盘或页式存储访问，而 Redis `Zset` 主要面向内存中的排序与范围操作。
