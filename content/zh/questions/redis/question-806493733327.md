## 答案

Redis 的哨兵（Sentinel）机制，本质上就是：
## Redis 官方提供的一套“高可用（HA）自动故障转移方案

它解决的问题很直接：
- Redis 主节点（Master）挂了怎么办？
- 谁来发现？
- 谁来切换新的主节点？
- 业务怎么自动恢复？

Sentinel 就是专门干这个的 ，如果没有 Sentinel Master 挂了 会导致 Redis 不会自动切换，Slave 不会自动锁升级，Java 项目连不上 Redis ，缓存直接雪崩 ， 登录状态失败等问题。 只能手动改配置， 手动切换主从， 重启服务。

**Sentinel 本身也是 Redis 进程** ，可以理解为“Redis 的监控 + 因为机器人” 。
**包含：**

- 监控 （监控 Master / Slave 是否存活）
- 自动故障转移 （Master 挂了自动选新 Master）
- 通知 （告诉其他 Sentinel）
- 配置中心 （告诉客户端新的 Master 地址）

一般 Sentinel 架构中**至少 3 个 Sentinel** ，Redis 主从部署Sentinel 独立运行。防止 Sentinel 自己挂掉，至少3个Sentinel原因是：如果为偶数，投票会困难。

**工作原理：**

1 ） **监控**： Sentinel 每秒会 ping Redis，如果：Master 没回复 ，Sentinel 会认为主观下线（SDOWN）。

2 ） **投票确认**：加入一个 Sentinel 说 “Master” 挂了，那是不够的，必须要多个 Sentinel 一起确认，比如： quorum = 2 （至少 2 个 Sentinel 认为 Master 挂了）才会进入客观下线（ODOWN）。这么做是为了防止网络抖动导致的误切换 。

3 ）**故障转移（最核心）**：Master 真的挂掉后，Sentinel 开始 Failover ，**第一步 选 Leader Sentinel** 。多个 Sentinel 先选举 ： 谁负责这次切换，避免大家一起改。 **第二步 选新的 Master** ：Sentinel 会从 Slave 中挑一个 。

挑选规则： 

- replication offset 最大 ；说明数据最新
- 网络最好 ； 优先选择
- 优先级高 ；replica-priority
-  响应快 ；延迟低
-  

**第三步 其他的 Slave 重新挂载** ：  ，原来的 `Slave2 -> old Master` 变成 `Slave2 -> new Master` ，随后执行 `SLAVEOF new-master`

**第四步 通知客户端 ：** Sentinel 会告诉客户端：新的 Master 地址 ，此处注意：客户端不是连 Redis ，而是：先连 Sentinel 获取当前的 Master ，所以 Master 变了也能自动感知 。

---

**缺点：**

1 ） 只能解决高可用 ，不能解决海量数据，因为它不是分片。
2 ） 主从切换期间会丢失数据 ，因为 Redis 主从采取异步复制，Master  还没同步给 Slave，Master 就挂了会丢失最后一部分数据。
3 ）切换期间短暂不可用： Failover 通常：几秒 ～ 几十秒 。 期间：写请求失败，锁失效，缓存不可写。

---

面试回答：

> **Redis Sentinel** 是 Redis 提供的高可用方案，主要用于监控 Redis 主从节点，并在 Master 宕机时自动完成故障转移。
Sentinel 会通过多个哨兵节点对 Master 做健康检测，当达到 quorum 数量后，会判定 Master 客观下线，然后由 Sentinel Leader 负责从 Slave 中选举新的 Master，并通知其他从节点重新同步，同时客户端也可以通过 Sentinel 自动发现新的主节点地址。
Sentinel 解决的是 Redis 的高可用问题，但不解决数据分片问题，因此大规模场景通常会使用 Redis Cluster。
