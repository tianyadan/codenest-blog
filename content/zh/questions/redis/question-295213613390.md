## 答案

## 简要回答

Redis 出现性能瓶颈时，应先定位瓶颈属于慢命令、热点或大 key、内存、网络、持久化还是容量不足，再针对性优化；单实例优化到上限后，再考虑读写分离或 `Redis Cluster` 分片扩容。

## 专业名词解释

### Big Key

Big Key 指占用内存很大，或集合成员数量过多的 key。它可能导致读写、删除、网络传输和主从同步耗时增加，引发延迟抖动。

### Hot Key

Hot Key 指访问量过度集中在少数 key 上的情况。即使总数据量不大，也可能因为单点请求压力过高而耗尽单个节点处理能力。

### 慢命令

慢命令是执行时间超过预期的 Redis 命令，例如对超大集合执行高复杂度操作，或使用 `KEYS` 扫描大量 key，会阻塞后续请求处理。

## 面试回答

Redis 出现性能瓶颈时，我不会直接先扩容，而是先通过指标和命令定位原因。首先观察**延迟**、**吞吐量**、**CPU**、**内存**、**网络**、**连接数**、**命中率**、**淘汰数量**以及主从复制延迟；再结合 `SLOWLOG GET`、`INFO`、延迟监控和采样分析，**确认是否存在慢命令**、**Big Key**、Hot Key、内存不足、持久化抖动或客户端使用不当。

如果是命令和数据模型问题，应避免 `KEYS`、大范围聚合等阻塞操作，改用 `SCAN` 分批处理；拆分 Big Key，删除大 key 时优先使用 `UNLINK`；对于 Hot Key，可增加本地缓存、拆分 key 或将读取压力分散。客户端侧还可以使用连接池、批量操作或 `Pipeline`，减少频繁建连和网络往返。

如果是内存问题，需要设置合理的 `maxmemory` 与淘汰策略，控制无效缓存和过期时间，选择更节省内存的数据结构，并避免系统发生 swap。若是 `RDB`、`AOF` 重写或主从同步引起抖动，则应调整持久化策略、预留足够内存与磁盘能力，并监控复制积压和延迟。

当单节点 CPU、内存或吞吐确实达到上限后，再进行架构扩展：读多写少场景可通过副本分担读请求；数据规模或写入吞吐持续增长时，可使用 `Redis Cluster` 将数据分片到多个主节点。核心原则是**先定位，再优化，最后扩容**，否则扩容可能只是暂时掩盖问题。

## 核心要点

- 先用监控和慢日志定位根因
- 优先排查慢命令、Big Key、Hot Key
- 避免阻塞操作，合理使用 `SCAN`
- 内存压力需控制淘汰策略和 swap
- 客户端可通过 `Pipeline` 降低往返开销
- 单节点到达上限后再分片扩容

## 面试追问

### 1. 如何排查 Redis 中的慢命令问题？

可以先通过 `SLOWLOG GET` 查看耗时较高的命令，再结合业务访问链路确认触发场景。重点排查 `KEYS`、大集合全量读取、复杂聚合操作和 Big Key 访问。发现问题后，应改为增量扫描、拆分结构、限制返回数据量或调整业务查询方式。

### 2. Big Key 和 Hot Key 分别会造成什么影响，如何处理？

Big Key 会放大单次命令、删除、复制和网络传输成本，通常通过拆分数据、限制集合规模、使用 `UNLINK` 异步删除处理。Hot Key 会导致访问集中到单个节点，可通过本地缓存、请求合并、key 分片或多副本分摊读流量处理。

### 3. 什么时候应该从单节点升级为 Redis Cluster？

当已完成命令、内存和客户端侧优化后，单节点仍长期面临内存不足、CPU 饱和或写入吞吐受限，并且业务数据可以按 key 分片时，可以考虑 `Redis Cluster`。它能水平扩展容量和吞吐，但会引入多 key 跨 slot、迁移和运维复杂度。

## 常见误区

**误区：** Redis 变慢时，直接增加机器配置或搭建集群就能解决问题。  
**正确理解：** 如果根因是慢命令、Big Key、Hot Key 或错误的数据结构，扩容只能缓解症状，甚至会放大运维成本，必须先定位并优化访问模式。

**误区：** Redis 核心命令执行很快，因此不会因为单个耗时命令影响其他请求。  
**正确理解：** Redis 的命令执行路径主要按顺序处理，阻塞型命令或超大 key 操作会拖慢同节点上的其他请求，造成明显延迟抖动。
