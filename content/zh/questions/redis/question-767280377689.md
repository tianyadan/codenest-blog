## 答案

## 简要回答

Redisson 的分布式锁基于 Redis key、Lua 脚本、可重入计数、发布订阅通知和 Watchdog 自动续期实现。它解决了加锁、解锁原子性及业务执行超时问题，但在主从切换等极端场景下仍需评估一致性风险。

## 专业名词解释

### 分布式锁

分布式锁用于协调多个进程或服务实例对同一共享资源的互斥访问，保证同一时刻只有符合条件的执行者进入临界区。

### 可重入锁

可重入锁允许同一个线程重复获取同一把锁，每次加锁增加重入次数，只有解锁次数匹配后锁才真正释放。

### Watchdog

Watchdog，即看门狗机制。Redisson 在线程持有未指定固定租约时间的锁时，会周期性延长锁的过期时间，避免业务尚未执行完毕时锁提前失效。

## 面试回答

Redisson 中常用的分布式锁对象是 `RLock`。加锁时，它通过 Lua 脚本在 Redis 中原子判断锁是否存在：如果不存在，就创建锁并设置过期时间；如果锁已经由当前线程持有，则增加重入计数并刷新过期时间；如果被其他线程持有，则返回剩余过期时间，当前线程进入等待。

锁的数据通常使用 `Hash` 保存，字段中包含客户端实例标识和线程标识，值表示重入次数，因此只有持锁线程才能正确解锁。解锁同样通过 Lua 脚本原子执行：先校验锁归属，再将重入次数减一；次数降为零时删除锁，并通过 Redis `Pub/Sub` 发布解锁消息，唤醒等待获取锁的线程，避免持续轮询。

Redisson 还提供 Watchdog 自动续期机制。调用未显式指定 `leaseTime` 的 `lock()` 成功后，只要持锁的 Redisson 实例仍正常运行，就会周期性延长锁过期时间；默认看门狗超时时间为 `30` 秒，可通过 `lockWatchdogTimeout` 配置。若显式指定 `leaseTime`，锁会在该租约到期后自动释放，通常不会依赖 Watchdog 无限续期。

使用时应在 `finally` 中解锁，并通过 `isHeldByCurrentThread()` 判断当前线程是否仍持有锁。还要注意，基于 Redis 主从异步复制的锁在故障切换窗口仍可能出现锁状态丢失，因此对资金、库存扣减等强一致场景，不能只依赖分布式锁，还需要数据库约束、幂等或业务补偿机制。

```java
RLock lock = redissonClient.getLock("lock:order:1001");
boolean locked = false;
try {
    locked = lock.tryLock(3, 10, TimeUnit.SECONDS);
    if (locked) {
        // 执行业务逻辑
    }
} finally {
    if (locked && lock.isHeldByCurrentThread()) {
        lock.unlock();
    }
}
```

## 核心要点

- `RLock` 支持互斥和可重入语义
- 加锁与解锁依赖 Lua 保证原子性
- 锁中会记录线程归属和重入次数
- `Pub/Sub` 用于通知等待线程重新竞争
- Watchdog 可为未固定租约的锁自动续期
- 强一致业务仍需配合幂等与数据约束

## 面试追问

### 1. Redisson 为什么能够实现可重入锁？

Redisson 会在锁对应的 Redis `Hash` 中记录客户端实例与线程标识，并保存该线程的重入次数。同一线程再次加锁时，不会被自己阻塞，而是将计数加一；解锁时逐次减一，只有计数归零才真正删除锁。

### 2. Watchdog 与显式指定 `leaseTime` 有什么区别？

未指定固定租约时间时，Redisson 会启用 Watchdog，在持锁实例存活期间自动续期，适合业务执行时间难以预估的场景。显式指定 `leaseTime` 后，锁到期会自动释放，适合执行耗时明确、希望限制最长占锁时间的业务。

### 3. Redisson 分布式锁是否可以完全保证业务不会重复执行？

不能。锁能降低并发冲突，但在客户端暂停、网络分区、Redis 故障切换或业务超时重试等情况下，仍可能出现边界问题。关键写操作应配合唯一约束、幂等键、状态机或数据库事务，不能把全部一致性责任交给锁。

## 常见误区

**误区：** 使用 Redisson 分布式锁后，业务就天然具备强一致性，不会发生重复处理。  
**正确理解：** Redisson 锁提供互斥控制和续期能力，但无法替代数据库一致性约束、幂等设计及异常补偿机制。

**误区：** 解锁时只要调用 `isLocked()` 判断锁存在，就可以安全执行 `unlock()`。  
**正确理解：** 锁存在不代表当前线程拥有它。应使用 `isHeldByCurrentThread()` 判断归属，避免误解锁其他线程后来获取到的锁。
