## 答案

## 简要回答

分布式锁在业务逻辑未执行完成前过期，会导致其他线程或节点重新获取锁，从而出现并发执行、数据覆盖或重复操作问题。常见解决方案是使用“锁续期”机制，也叫看门狗机制。

## 专业名词解释

### 看门狗机制

看门狗机制是指在业务执行期间，后台线程定时为锁延长过期时间，避免业务未完成时锁提前失效。`Redisson` 默认内置了该机制。

### 锁续期

锁续期是指在锁即将过期时，主动刷新锁的 TTL，使锁继续保持有效状态，从而保证长时间任务执行期间不会被其他线程抢占。

## 面试回答

分布式锁如果在业务逻辑执行完成前过期，会导致锁失效，其他线程或服务节点可能重新获取到同一把锁，从而出现并发执行问题。例如重复扣库存、重复下单、数据覆盖等。

这个问题通常出现在业务执行时间不可控的场景，比如远程调用、IO 阻塞、大批量处理或者 Full GC 导致线程暂停。

业界最常见的解决方案是引入“锁续期”机制，也就是看门狗机制。比如 `Redisson` 的 `lock.lock()` 默认会启动后台续期线程，在业务未结束时不断延长锁的过期时间，避免锁提前失效。

如果不使用自动续期，也可以根据业务最大执行时间手动设置较长过期时间，但这种方式不够灵活：时间太短可能失效，时间太长又会影响锁释放效率。

另外，释放锁时一定要校验锁的持有者身份，通常会使用 UUID 配合 Lua 脚本，避免误删其他线程重新获取到的锁。

```bash
SET order_lock uuid123 NX EX 30
```

安全释放锁：

```lua
if redis.call('get',KEYS[1]) == ARGV[1]
then
    return redis.call('del',KEYS[1])
else
    return 0
end
```

## 核心要点

- 锁提前过期会导致并发问题
- 长耗时业务容易触发锁失效
- 看门狗机制可自动续期
- `Redisson` 内置锁续期能力
- 释放锁必须校验持有者
- Lua 脚本可避免误删锁

## 面试追问

### 1. 为什么不能把锁过期时间设置得特别长？

锁时间过长虽然能避免提前失效，但如果服务异常宕机，锁会长时间无法释放，影响系统吞吐量，还可能导致后续请求一直阻塞。

### 2. `Redisson` 的看门狗机制是怎么工作的？

`Redisson` 获取锁后会启动后台定时任务，默认每隔一段时间自动刷新锁的过期时间。只要业务线程还持有锁，锁就不会失效；业务结束后续期线程也会停止。

### 3. 为什么释放锁时必须校验 UUID？

因为锁过期后可能已经被其他线程重新获取。如果当前线程直接 `DEL key`，就可能把别人刚获取到的锁删掉，导致锁失效和并发安全问题。

## 常见误区

**误区：** 只要设置了 Redis 锁过期时间，就一定安全。  
**正确理解：** 如果业务执行时间超过锁 TTL，锁仍然可能提前失效，需要结合续期机制或合理 TTL 控制。

**误区：** 锁释放时直接 `DEL key` 就可以。  
**正确理解：** 必须校验锁持有者身份，否则可能误删其他线程已经重新获取到的锁。
