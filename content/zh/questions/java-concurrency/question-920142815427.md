## 答案

**线程池是一种池化技术，用于预先创建并管理一组线程。**

**总结一句话：**

Java 线程池本质上是一个**任务调度中心**，通过**线程复用** + **阻塞队列** + **原子状态管理**，在性能与稳定性之间找到最佳平衡点。

**为什么使用线程池？**
如果直接使用 new Thread() 的方式创建线程会导致频繁创建和销毁线程，线程数量不可控制，缺乏统一管理。

因此为了**避免频繁创建**和**销毁线程**的开销，**提高性能和响应速度**。 线程池的目标是实现复用线程、限制并发数、统一调度任务。

**线程池核心原理：**

Java 的线程池主要由 ThreadPoolExecutor 实现，它的工作原理可以概括为：“先用已有线程执行任务，不够再创建新线程，满了就排队列，排队也满了就拒绝。”

**完整任务流程：**

1. **提交任务** 

外部线程调用：


```java
execute( Runnable task )
```

或者

```java
submit( Callable / Runnable )
```

其中：

- execute（）没有返回值
- submit （）会返回 true / false
- 底层核心人物仍是把任务丢给线程池执行


2.  **判断当前工作线程数是否小于 corePoolSize**

如果：

```java
workerCount < corePoolSize
```

线程池会优先创建核心线程执行当前任务，而不是先放队列。

也就是说：核心线程没满时，任务来了先加线程，不进队列。

 
3. **核心线程已满，尝试放入阻塞队列**

如果当前工作线程数已经达到 corePoolSize，线程池会尝试把任务放进 workQueue。如果队列没满：`workQueue.offer(task) == true`，任务进入队列，等待空闲线程来取。

注意：
这里用的是 offer()，不是一直阻塞等待。

4. **入队后会再次检查线程池状态**

这是很多人会漏掉的点。任务成功入队后，线程池还会再检查一次状态：    如果线程池**已经关闭**了，就把任务从队列移除，**并执行拒绝策略**，如果线程池还在运行，但当前没有工作线程，就创建一个线程来消费队列任务。

这个是为了避免：任务进了队列，但没有线程处理它。


5. **如果队列满了，尝试创建非核心线程**

如果任务放入队列失败，说明队列满了。这时线程池会判断：`workerCount < maximumPoolSize`，如果成立，就创建新的非核心线程来执行当前任务。注意这里执行的是：当前新提交的任务，不一定是队列里的旧任务。

6. **如果最大线程数也满了，触发拒绝策略**

如果：`workerCount >= maximumPoolSize`并且队列也满了，就会触发拒绝策略：`RejectedExecutionHandler` 

常见拒绝策略：
- AbortPolicy：直接抛异常，默认策略
- CallerRunsPolicy：由提交任务的线程自己执行
- DiscardPolicy：直接丢弃任务
- DiscardOldestPolicy：丢弃队列中最老的任务，再尝试提交新任务

7. **工作线程执行任务**

线程池里的 Worker 会循环执行任务：


```java
while ((task = getTask()) != null) {
    task.run();
}
```

它的逻辑是：
- 先执行创建线程时绑定的第一个任务
- 执行完后，不马上销毁
- 继续从阻塞队列中取任务执行

所以线程池不是“一个线程只执行一个任务”，而是：一个 Worker 线程会不断复用，循环处理多个任务。

8. **空闲线程回收**

当线程空闲超过：`keepAliveTime` 线程池会回收线程。默认情况下：核心线程不会被回收 ，非核心线程会被回收 。但如果设置：`allowCoreThreadTimeOut(true)` ，那么核心线程空闲超过 keepAliveTime 后，也会被回收。


**总结 ：**
> ThreadPoolExecutor 提交任务后，首先判断当前工作线程数是否小于 corePoolSize，如果小于，就创建核心线程执行任务。
> 如果核心线程已满，就尝试把任务放入阻塞队列。
> 如果队列也满了，再判断当前线程数是否小于 maximumPoolSize，如果小于，就创建非核心线程执行当前任务。
> 如果线程数已经达到 maximumPoolSize，就触发拒绝策略。
> 工作线程执行完任务后不会立即销毁，而是循环从队列中获取任务继续执行。空闲超过 keepAliveTime 的非核心线程会被回收；如果开启 allowCoreThreadTimeOut，核心线程也会被回收。

**核心参数**

ThreadPoolExecutor 的构造函数：


```Java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    corePoolSize,        // 核心线程数
    maximumPoolSize,     // 最大线程数
    keepAliveTime,       // 空闲线程存活时间
    TimeUnit.SECONDS,    // 时间单位
    workQueue,           // 阻塞队列
    threadFactory,       // 线程工厂（可自定义线程名）
    handler              // 拒绝策略
);
```

---

**常见队列类型：**
 
| 类型 | 类名 | 特点 |
| --- | ---| ---  |
| 有界队列| ArrayBlockingQueue | 限制队列大小，常用 （推荐优先使用） |
| 无界队列（链表队列）| LinkedBlockingQueue | 默认最大值Integer.MAX_VALUE, 默认线程池很多用这个 → 容易 OOM |
| 同步队列| SynchronousQueue | 不存任务，直接交给线程处理（常用于缓存线程池），特点是直接触发扩容线程|
| 优先级队列| PriorityBlockingQueue |按优先级执行，需要实现 Comparable |
| 延迟队列| DelayQueue |     定时任务，延迟执行 |

---

**线程工厂(ThreadFactory):**

1 ) 默认 Executors.defaultThreadFactory()

 特点：线程名:pool-thread-1 （不好排查问题）
 

2 ）自定义 （推荐）


```java
ThreadFactory factory = r -> {
    Thread t = new Thread(r);
    t.setName("my-pool-" + t.getId());
    return t;
};
```

特点：

- 可设置线程名
- 是否守护线程
- 优先级

3 ）高级用法（生产推荐）

用 Guava ThreadFactoryBuilder：


```java
new ThreadFactoryBuilder()

    .setNameFormat("biz-pool-%d")

    .setDaemon(false)

    .build();
```

优点：

- 线程名清晰
- 好排查日志


---

**拒绝策略（超重点，必问）**
 
当任务太多时，线程池会拒绝任务，有四种内置策略。


| 策略 | 说明 |
| --- | --- |
|  AbortPolicy  |  默认，抛异常，最严格 ， 不丢任务，但会报错|
|  CallerRunsPolicy（推荐）  | 让提交任务的线程自己执行，即谁提交谁执行，可用于降低系统吞吐，保护线程池 |
|  DiscardPolicy（极不推荐）  |  直接丢弃任务 ， 容易导致数据丢失|
|  DiscardOldestPolicy  |  丢弃队列最老的任务，执行新任务，适合实时性要求高的系统|

---

**线程池类型：**

```Java
Executors.newFixedThreadPool(n) // 固定大小线程池
Executors.newCachedThreadPool(); // 缓存线程池 （无界）
Executors.newSingThreadExecutor(); //单线程池
Executors.newScheduledThreadPool(n); // 定时任务线程池
```


---

**总结：**

生产环境线程池参数选择： 

1 ） 阻塞队列： 优先 ArrayBlockingQueue ，避免 OOM
2 ） 线程工厂：自定义命名，方便排查问题。
3 ） 拒绝策略：使用 callerRunsPolicy 做限流保护。
4 ） 线程数： 根据 CPU/IO 类型调整
