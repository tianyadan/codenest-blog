## 答案

synchronized 实现原理依赖于 JVM 的 Mointor（监视器锁）和对象头 (Object Header)。

当 synchronized 修饰在方法或代码块上时，会对特定的对象或类加锁，从而确保同一时刻只有一个线程能执行加锁的代码快。

- synchronized 修饰方法：会在方法的访问标志中增加一个 ACC_SYNCHRONIZED  标志 。每当一个线程访问该方法时，JVM 会检查方法的访问标志。如果包含 ACC_SYNCHRONIZED  标志，线程必须先获得该方法对应的对象的监视器锁（即对象锁），然后才能性。

- synchronized 修饰代码块：会在代码块的前后插入 mointorenter 和 monitorexit 字节码指令。可以把**monitorenter 理解为加锁** ，  **monitorexit 理解为解锁** 。


**扩展知识：**

**对象头（Object Header）**

 在 JVM 中，每个对象的内存布局主要由两部分组成：
 
 - Mark Word ： 用于存储对象的运行时数据，包括锁状态，哈希码，GC 分代信息等。
 - Klass Pointer ：指向对象的类型元数据，帮助 JVM 确定对象的类型信息。
 
**Mark Word** 是实现 synchronized 的关键，因为它会根据锁的状态保存不同的信息，具体包括：
 
 - 未锁定状态（这是对象刚出生、没人碰它时的状态）：
Mark Word 存储对象的哈希码和 GC 分代信息（GC 年龄（对象多少次晋升））。

- 偏向锁状态（反正就你一个人用这个对象锁，那以后都属于你，不用再抢了）：
 Mark Word 保存获取该锁的偏向线程 ID、Epoch（偏向锁版本）、偏向标志位（biased flag）。

- 轻量级锁状态（当第二个线程也想进入 synchronized(obj) 时，就无法保持偏向锁了）： 
JVM 会把偏向锁升级为轻量级锁，每个线程在自己的栈帧里创建一个 Lock Record（锁记录），然后 CAS 尝试把 Mark Word 变成指向这个 Lock Record 的指针
轻量级锁的特点：不会阻塞线程、自旋等待、CAS 反复尝试加锁

- 重量级锁状态（如果竞争激烈，CAS 一直失败）： Mark Word 存储的是指向 Monitor 对象的指针。

🎯**一句话总结：**
Mark Word 是 synchronized 的“大脑” ，它根据不同锁状态存不同的数据来驱动锁升级：
无锁 → 偏向锁 → 轻量级锁 → 重量级锁

通俗易懂版本：
	•	无锁：空房子，没人租。
	•	偏向锁：一个固定租户，给他配了门禁卡，他随便进出。
	•	轻量级锁：来了第二个租户，门口贴了“先来后到”登记表（Lock Record），大家自觉排队。
	•	重量级锁：实在太多人争抢出租屋，房东叫来保安（Monitor），统一管理排队。
