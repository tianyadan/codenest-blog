## 答案

回答：

**不可以。**

一般而言我们值得可见性是一个线程修改了共享变量，另一个线程可以立马得知更改，得到最新修改后的值。

而 **final** 并不能保证这种情况发生，**volatile** 才可以。

示例：
```Java
public class YesFinalTest {
   final int a; 
   int b;
   static YesFinalTest testObj;

   public YesFinalTest () { //对字段赋值
       a = 1;
       b = 2;
   }

   public static void newTestObj () {  // 此时线程 A 调用这个方法
       testObj = new YesFinalTest ();
   }

   public static void getTestObj () {  // 此时线程 B 执行这个方法
       YesFinalTest object = testObj; 
       int a = object.a; //这里读到的肯定是 1
       int b = object.b; //这里读到的可能是 2
   }
}
```

### 结论：

- 线程 B 一定能看到 a = 1；

**原因**：构造函数内对 final 字段的写入，会在构造函数“结束前”，把这些写入同步到主内存。

更直白地说：

JVM 会禁止这两种重排序：
	1.	对象引用（testObj）对外可见 不能 发生在 final 字段写入之前
	2.	final 字段写入不能被调出构造函数外执行

这就形成一个保证：

👉 只要构造函数执行完，其他线程一定能看到 final 字段的正确值。

所以线程 B 即使发生了指令重排，只要它拿到 testObj，一定能看到：



- b 有可能不是 2 ；

**原因：**

因为 b 是普通字段。

普通字段是这样：

JMM 不保证构造函数对普通字段的写入一定“先于”对象引用对外发布。

可能发生如下重排序（极端情况举例）：


```Java
1. testObj = new YesFinalTest(); // 分配内存 + 未完全构造
2. 另一个线程 B 读到 testObj
3. 线程 B 访问 object.b  -> 此时 b 还是默认值 0
4. 线程 A 才把 b 写成 2
```

所以：
👉 B 读取到的 b，可能是 0，也可能是 2。

虽然在实践中非常罕见，但 JMM 允许这样发生，面试就爱问这种“罕见但必考”的点。
