## 答案

Spring IOC （Inversion of Control） 即控制反转，是 Spring 框架的核心概念之一，它是通过**依赖注入(Dependency Injection)** 实现的。IOC 让对象的创建**由容器负责**而不是由对象自身控制。

核心思想：控制反转意味着将对象的创建与依赖关系**由 Spring 容器**管理，而不是由程序代码直接控制。这种机制使得程序**更加灵活和解耦**，提升代码可维护性和扩展性。

所谓**控制**，到底是在控制什么？ --**控制对象的创建**，不同时期根据不同配置进行对象的创建和改造。

所谓**反转**，又是在反转什么？ --例如我在编程时考虑到 A 对象依赖于 B 对象，那么在手写创建对象 A 的 code 中要写好如何创建 B 对象，这样 A 才能完整构建。

**但是**有了控制反转之后这个动作由 IOC 容器触发，它发现创建 A 对象时需要依赖 B 对象，那么此时它会先创建 B 对象，然后注入到 A 中，此过程就是反转。

🏠 场景比喻：自己租房 vs 找中介
- 	没 IOC 的情况（自己写代码 new 对象）：
你要租房（创建 A 对象），结果房东（A）还需要提供床、桌子、沙发（B、C、D）。
没有 IOC 的时候，你得 自己跑市场，先买床（new B）、再买桌子（new C）、再买沙发（new D），最后才算住进去。
👉 你自己负责“依赖关系”的管理。


- 	有 IOC 的情况（Spring 容器）：
你只告诉中介（Spring 容器）：“我需要一个能住的房子（我要 A 对象）”。
中介会自动帮你准备好床、桌子、沙发（把 B、C、D 都 new 好），然后一股脑塞进房子里。
👉 你只管要，容器负责创建和注入依赖。

传统做法（正向控制）：得先把 B 创建好，再把它塞到 A 里。


```
B b = new B();
A a = new A(b);
```

IOC（反转控制）：

```
@Autowired
private B b;
```

啥也不管，只在 A 上写个注解。
Spring 容器在启动时就会发现：
- 	“哦，A 需要一个 B”
- 	它就先 new 出 B，再把 B 放进 A 里
- 	最后交给你一个完整的 A

👉 反转的点就是：
“控制对象依赖的主动权”从 开发者 手里，反转到了 容器（Spring） 手里。
